# 1. relation抽出の基本方針

Rwikiでは relation を最初から「正解」として扱わない方がよいです。

## relation = 仮説

として扱う。

つまり：

```text
A relates_to B
```

ではなく、

```text
A may_relates_to B
confidence: 0.72
evidence: source.md の該当箇所
status: candidate
```

にする。

これが重要です。

------

# 2. 抽出対象は3層に分ける

いきなり複雑な関係を全部取ると壊れます。

## Layer 1：明示的relation

本文に直接書かれている関係。

例：

```text
AはBの原因である
AはBを改善する
AはBを用いる
AはBに依存する
```

これは比較的安全。

------

## Layer 2：近接・共起relation

同じ段落・同じ実験・同じ議論に出たもの。

例：

```text
A and B appear in same context
A is discussed_with B
```

これは弱い関係。

------

## Layer 3：推論relation

LLMが意味的に推定する関係。

例：

```text
A may_explain B
A is_alternative_to B
A is_tradeoff_with B
A is_generalization_of B
```

これは便利だが危険。

------

# 3. 最初に使うべきrelation type

最小実装では、relation typeを増やしすぎない方がよいです。

おすすめはこの10個程度です。

```yaml
relation_types:
  - uses
  - depends_on
  - causes
  - improves
  - degrades
  - compares_with
  - alternative_to
  - part_of
  - supports
  - contradicts
```

加えて、弱い関係として：

```yaml
  - co_occurs_with
  - related_to
```

を置く。

ポイントは、**最初から立派なontologyを作らない**ことです。

------

# 4. 抽出アルゴリズムの流れ

## Step 1：Entity抽出

まず本文から重要概念を取る。

```text
入力文書
↓
重要Entity抽出
↓
正規化
↓
Entity候補リスト
```

例：

```yaml
entities:
  - Rwiki
  - GraphRAG
  - relation extraction
  - approve flow
  - input cost
```

------

## Step 2：Relation候補抽出

Entityペアに対して関係を抽出する。

```yaml
relations:
  - source: Rwiki
    relation: has_problem
    target: input cost
    confidence: 0.92
    evidence: "Rwikiは...人間がかなり手を入れる必要がある"
```

ただし `has_problem` のような自由relationを許すと増殖するので、内部では既定typeに正規化する。

```yaml
relation: suffers_from
```

ではなく、

```yaml
relation: depends_on
# または
relation: degrades
```

のように寄せる。

------

## Step 3：Evidence必須化

relationには必ず根拠を付ける。

```yaml
evidence:
  file: raw/logs/xxx.md
  quote: "GraphRAG以前にGraphが育たない"
  span: line 120-124
```

これがない relation は採用しない。

------

## Step 4：Confidence付与

confidenceは単なるLLMの自己評価では弱いので、複数要素で決める。

```text
confidence =
  明示度
+ evidenceの強さ
+ 既存graphとの整合性
+ 複数sourceでの再出現
- contradiction
```

初期実装では単純でよいです。

```yaml
confidence:
  explicit: 0.9
  co_occurrence: 0.4
  inferred: 0.6
```

------

核心は、**confidenceを「LLMの自信」ではなく「証拠の強さ」として計算する**ことです。

私ならこうします。

# Confidence = relationの採用優先度

まず confidence は「真偽確率」ではなく、

> この relation をGraphに入れて使ってよい度合い

と定義します。

------

# 1. 基本式

```text
confidence =
  0.35 * evidence_score
+ 0.20 * explicitness_score
+ 0.15 * source_reliability
+ 0.15 * graph_consistency
+ 0.10 * recurrence_score
+ 0.05 * human_feedback
```

初期実装ではこれで十分です。

------

# 2. 各スコアの意味

## evidence_score

根拠引用がどれだけ直接的か。

```text
0.9〜1.0 明示的に書いてある
0.6〜0.8 文脈からかなり自然
0.3〜0.5 共起しているだけ
0.0〜0.2 根拠が弱い
```

最重要です。

------

## explicitness_score

関係語が本文にあるか。

例：

```text
AはBを引き起こす
AはBに依存する
AはBと比較される
```

なら高い。

```text
同じ段落にAとBがある
```

なら低い。

------

## source_reliability

入力元の信頼度。

```text
reviewed paper        0.9
approved synthesis    0.9
user-authored note    0.8
discussion log        0.6
auto summary          0.5
web clip              0.4〜0.7
```

Rwikiでは、**raw log由来relationは低め、synthesis由来relationは高め**がよいです。

------

## graph_consistency

既存Graphと矛盾しないか。

```text
既存edgeと補強関係      +高
同じクラスタ内          +中
既存edgeと矛盾          -大
孤立                   +低
```

ただし、矛盾を即削除しない。
`contradicts` として保持する。

------

## recurrence_score

複数sourceで同じ関係が出たか。

```text
1 source     0.2
2 sources    0.5
3+ sources   0.8
独立source多数 1.0
```

重要なのは、**同じ会話ログ内の繰り返しを過大評価しない**ことです。

------

## human_feedback

人間が明示的に触ったか。

```text
rejected         0
edited           0.7
accepted         1.0
used_in_answer   +0.1〜0.2
```

------

# ■ 2. 役割の本質

Graph Hygieneの役割はこれです：

> **「正しさ」ではなく「生き残りやすさ」を決める**

つまり：

- 正しいが使われない → 消える
- 少し間違ってても使われる → 残る

👉 かなり重要な思想転換です

------

# ■ 3. 3つの基本ダイナミクス

Graph Hygieneは最低限この3つで構成します。

------

## ① Decay（忘却）

使われないrelationは弱くなる。

```text
confidence(t+1) = confidence(t) * (1 - λ)
```

例：

```text
λ = 0.01 / day
```

👉 「放置＝削除」に近づく

------

## ② Reinforcement（強化）

使われたrelationは強くなる。

```text
confidence += α * usage_signal
```

usage_signal例：

- queryで使用
- explanationで採用
- 他relationの推論に使われた

👉 **queryが学習になる**

------

## ③ Competition（競争）

似たrelation同士が競う。

例：

```text
A → B (causes)
A → B (correlates_with)
```

→ どちらかが優勢になる

```text
confidence_i = confidence_i / Σ(similar_relations)
```

👉 Graphの冗長性が減る

------

# ■ 4. さらに重要な2つ（ここが肝）

------

## ④ Contradiction tracking（矛盾管理）

矛盾は削除しない。

```yaml
A causes B
A does_not_cause B
```

↓

```yaml
relation:
  type: contradicts
```

そして：

```text
両方のconfidenceを少し下げる
```

👉 真理判定ではなく「未確定状態」を保持

------

## ⑤ Merging（同一化）

似たrelationを統合する。

```text
improves ≒ enhances ≒ increases
```

↓

```text
improves に吸収
```

👉 ontologyを後から自然形成

------

# ■ 5. 実際の更新式（シンプル版）

1ステップでやるなら：

```text
confidence_next =
  confidence_current
  + α * usage
  + β * recurrence
  - γ * decay
  - δ * contradiction
```

初期値：

```text
α = 0.05
β = 0.03
γ = 0.01
δ = 0.1
```

------

# ■ 6. 状態遷移

relationは状態を持たせるとよいです。

```yaml

  - weak

  - stable
```

```text

weak → (さらに放置) → deprecated
```

👉 Graphが「呼吸」する

ここから一気に“動く設計”に入れます。



ここはかなり重要で、設計を一歩間違えると
**「人気＝真実」になってGraphが壊れます。**

結論から言うと：

> **usage_signalは「使われた回数」ではなく
> 「推論にどれだけ貢献したか」で定義するべきです。**

------

# ■ 1. usage_signalの基本定義

まずシンプルに書くと：

```text
usage_signal = 貢献度 × 信頼度 × 独立性
```

------

# ■ 2. 「使われた」の種類を分解する

usageは1種類ではダメです。最低でも4種類に分ける。

------

## ① Direct Usage（直接使用）

relationが回答生成に直接使われた

例：

```text
Q: Rwikiの問題は？
A: 入力コスト
```

内部：

```text
Rwiki → has_problem → input_cost
```

👉 最も強い

```text
score = 1.0
```

------

## ② Support Usage（補助的使用）

論理展開の一部で使われた

例：

```text
input_cost → degrades → graph_growth
```

👉 中程度

```text
score = 0.6
```

------

## ③ Retrieval Usage（検索ヒット）

検索にはヒットしたが、回答には使われなかった

👉 弱い

```text
score = 0.2
```

------

## ④ Co-activation（同時活性）

同じqueryで一緒に呼ばれた

👉 ノイズ多め

```text
score = 0.1
```

------

# ■ 3. 重要：貢献度の重み付け

usage_signalの核心はここです。

------

## 貢献度 = reasoningでの役割

```text
usage_contribution =
  answer_core_weight * is_core
+ chain_weight * position_in_chain
```

例：

```text
最終結論に直結 → 1.0
中間推論       → 0.6
周辺情報       → 0.3
```

------

# ■ 4. 信頼度補正

低confidenceのrelationが使われても強化しすぎない。

```text
usage_signal *= sqrt(confidence)
```

👉 低品質relationの暴走防止

------

# ■ 5. 独立性（超重要）

これを入れないと壊れます。

------

## 問題

同じソースから派生したrelationが何度も使われる：

👉 偽の強化

------

## 解決

```text
independence_factor =
  1 / (同一sourceからの使用回数)
```

または：

```text
distinct_sources_count を使う
```

------

# ■ 6. 時間減衰（Temporal weighting）

最近のusageを優先：

```text
time_weight = exp(-Δt / τ)
```

例：

```text
τ = 7日
```

------

# ■ 7. 最終式

まとめると：

```text
usage_signal =
  base_usage_score
  × contribution_weight
  × sqrt(confidence)
  × independence_factor
  × time_weight
```

------

# ■ 8. 例

```yaml
relation: Rwiki → has_problem → input_cost
```

### ケース1：コア回答で使用

```text
base_usage_score = 1.0
contribution = 1.0
confidence = 0.8 → sqrt = 0.89
independence = 1.0
time_weight = 1.0

usage_signal ≈ 0.89
```

------

### ケース2：補助的に使用

```text
base = 0.6
contribution = 0.6
confidence = 0.7 → 0.84
independence = 0.5
time_weight = 0.9

usage_signal ≈ 0.14
```

# ■ 9. どう記録するか（実装）

relationごとにこれを持つ：

```yaml
usage:
```

usage_signal設計で一番やってはいけないこと：

------

## ❌ NG

```text
使われた回数 += 1
```

------

## ✅ 正解

```text

```

# ■ 11. Rwiki的に重要な補正

Rwikiではさらにこれを入れると強い：

------

## ▶ Evidence再利用ボーナス

同じevidenceが複数回使われる：

```text
usage_signal += 0.1
```

👉 「引用される事実」が強化される

## ▶ Cross-contextボーナス

異なる文脈で使われる：

```text

```

👉 汎用知識が強くなる

# ■ 12. 本質

usage_signalは：

```text
「このrelationがどれだけ思考に使われたか」
```

であって、

```text
「どれだけ頻出したか」
```

ではない。

# ■ 13. 次に来る核心

ここまで来ると、次はこれです：

👉 **competitionをどの粒度でやるか**

- 同一nodeペアだけ？
- relation typeも含める？
- embedding距離でやる？

ここでGraphの形が決まります。

行きます？



では **competitionの粒度**です。
結論から言うと、最初は **3段階 competition** がよいです。

# 1. Level 1：同一ノードペア内 competition

最小・必須です。

```text
A → B
```

に対して複数relationがある場合：

```text
A causes B
A supports B
A correlates_with B
A discusses B
```

これらを競わせる。

目的は、**同じA-B間で関係ラベルが増殖するのを防ぐ**こと。

------

# 2. Level 2：類似ノードペア competition

次に必要です。

例：

```text
Rwiki → input cost
Rwiki system → input burden
Rwiki workflow → curation cost
```

これらは実質的に近い。

ここではノード正規化・merge候補を出す。

目的は、**同じ概念が別名で増殖するのを防ぐ**こと。

------

# 3. Level 3：意味的に競合するrelation competition

これは高度です。

例：

```text
manual curation improves graph quality
manual curation degrades scalability
```

これは矛盾ではなく、**トレードオフ**です。

この場合、片方を潰さず、

```text
manual curation --improves--> graph quality
manual curation --degrades--> scalability
manual curation --has_tradeoff--> quality_vs_scalability
```

のように整理する。

目的は、**矛盾・対立・トレードオフを知識として残す**こと。

------

# 推奨実装順

最初はこれで十分です。

```text
Phase 1:
  同一node pairだけcompetition

Phase 2:
  類似node merge候補

Phase 3:
  tradeoff / contradiction整理
```

------

# competition score

各relationに勝敗をつけるなら：

```text
competition_score =
  confidence
+ usage_strength
+ recurrence
+ human_feedback
- redundancy_penalty
```

ただし、削除ではなく状態変更にします。

```text
winner      → stable
runner-up   → candidate
loser       → weak
obsolete    → deprecated
```

------

# 重要な設計原則

**competitionは「正解を1つに決める処理」ではありません。**

Rwikiではこう捉えるべきです。

```text
competition = graphを圧縮・整理する処理
```

つまり：

- 似たものをまとめる
- 弱いものを下げる
- 矛盾を明示する
- tradeoffを発見する

------

# 最小仕様に落とすなら

```yaml
competition:
  scope:
    - same_node_pair
    - similar_node_pair_later
    - semantic_conflict_later

  action:
    - promote
    - demote
    - merge_candidate
    - mark_tradeoff
    - mark_contradiction

  never:
    - hard_delete_without_log
```

これで、Graphが汚れながらも自浄作用を持ちます。



適切なのは、最初から本格Graph DBにせず、

**Git管理できるファイルGraphを正本にし、必要に応じてSQLite/NetworkX等で派生インデックスを作る方式**です。

# 結論

Rwikiではこうするのがよいです。

```text
Markdown / YAML / JSONL = 正本
SQLite / NetworkX / Graph DB = 派生キャッシュ
```

つまり、Graph DBを正本にしない。

------

# 1. なぜGraph DB正本にしないか

Neo4jなどを最初から正本にすると：

- Gitで差分管理しにくい
- relationの発生根拠を追いにくい
- 人間がレビューしにくい
- Rwikiの「軽量・可搬」思想とズレる
- confidence更新履歴がブラックボックス化しやすい

ので、初期段階では重いです。

------

# 2. 推奨ディレクトリ構成

```text
wiki/
  graph/
    entities.yaml
    edges.jsonl
    edge_events.jsonl
    rejected_edges.jsonl
    merge_candidates.yaml
    graph_stats.yaml
```

特に重要なのは `edges.jsonl` と `edge_events.jsonl` です。

------

# 3. edgeはJSONLがよい

YAMLは人間には読みやすいですが、大量relationには向きません。
edge本体はJSONLがよいです。

```json
{"edge_id":"e_001","source":"Rwiki","type":"depends_on","target":"auto_relation_extraction","confidence":0.82,"status":"stable","evidence_ids":["ev_001"],"created_at":"2026-04-24","updated_at":"2026-04-24"}
```

1行1edgeにすると：

- appendしやすい
- diffしやすい
- スクリプト処理しやすい
- 壊れにくい

------

# 4. confidence更新はevent logで管理

confidenceを直接上書きするだけだと、なぜ上がったか分からなくなります。

なので：

```text
edges.jsonl        = 現在状態
edge_events.jsonl  = 更新履歴
```

に分ける。

例：

```json
{"edge_id":"e_001","event":"used_in_query","delta":0.04,"query_id":"q_023","reason":"core_answer","timestamp":"2026-04-24"}
{"edge_id":"e_001","event":"decay","delta":-0.01,"timestamp":"2026-04-25"}
{"edge_id":"e_001","event":"human_reject","delta":-1.0,"timestamp":"2026-04-26"}
```

これはかなり重要です。

------

# 5. Evidenceはedgeから分離する

relation本体に長い引用を埋め込むと重くなります。

```text
evidence.jsonl
```

を別に持つのがよいです。

```json
{"evidence_id":"ev_001","file":"raw/logs/2026-04-24.md","quote":"relation抽出はデフォルトONに近いUX","span":"L20-L24"}
```

edge側は evidence_id だけ参照します。

------

# 6. 実装モデル

最小構成はこれです。

```text
正本:
  entities.yaml
  edges.jsonl
  evidence.jsonl
  edge_events.jsonl

処理:
  rw graph build
  rw graph score
  rw graph hygiene
  rw graph query

派生:
  .rw/cache/graph.sqlite
  .rw/cache/networkx.pkl
```

------

# 7. SQLiteをキャッシュに使う

検索やquery時にはJSONLを毎回読むと遅いので、SQLiteにビルドします。

```text
edges.jsonl → graph.sqlite
```

SQLiteには：

```text
nodes
edges
evidence
events
```

を入れる。

ただし、SQLiteは正本ではなくキャッシュ。

------

# 8. query時の流れ

```text
user query
  ↓
vector検索
  ↓
関連entity抽出
  ↓
graph.sqliteで近傍探索
  ↓
confidence/statusでfilter
  ↓
answer生成
  ↓
used edgeをedge_events.jsonlに追記
  ↓
次回hygieneでconfidence更新
```

------

# 9. status設計

edgeには状態を持たせます。

```text
weak
candidate
stable
core
deprecated
rejected
```

queryで使うのは基本：

```text
stable + core + candidate上位
```

`weak` は探索用、`deprecated/rejected` は通常使わない。

------

# 10. 最初の実装方針

最初はこれで十分です。

```text
Phase 0:
  edges.jsonl + evidence.jsonl + simple scorer

Phase 1:
  SQLite cache + query近傍探索

Phase 2:
  usage event + confidence更新

Phase 3:
  competition / merge / contradiction

Phase 4:
  必要ならNeo4j等へexport
```

------

# 一番大事な設計判断

RwikiではGraphを、

```text
巨大なGraph DB
```

ではなく、

```text
証拠付きedgeの台帳
```

として実装するのがよいです。

これなら、Rwikiの思想である

- Gitで管理できる
- 人間が読める
- 差分が追える
- 根拠に戻れる
- LLMが更新できる

を保ったまま、GraphRAG的に育てられます。

~~~
❯ Layer1からLayer3までの処理の流れを説明してください

⏺ Rwiki v2 の 3 層アーキテクチャでの処理の流れを 入力（forward） と 活用（reverse） の両方向で説明します。

  全体像

  [人間] ──書く──► L1 Raw ──LLM 抽出──► L2 Graph Ledger ──人間承認──► L3 Curated Wiki
                    raw/**/*.md          .rwiki/graph/*.jsonl         wiki/**/*.md
                                                ↑                            │
                                                └────── query / perspective ◄┘
                                                         （L2 を traverse、結果を返す）

  各層の責務:

 
│      層       │       物理位置      │           役割                │   人間関与   │
│L1 Raw         │raw/**/*.md         │ 入力、evidence 源             │ 入力のみ     │
│L2 Graph Ledger│.rwiki/graph/*.jsonl│ edge/entity/evidence の正本、 │ reject only │
│               │  + *.yaml          │ Hygiene で自己進化            │             │
│L3 Curated Wiki│wiki/**/*.md        │ 人間承認済 markdown           │ approve 必須 │

  ---
  Forward 方向: 入力 → 知識化

  Step 1: L1 Raw への入力

  ユーザーが raw/incoming/ に markdown を投入（手書き / Obsidian / Web clipper / Zotero export 等）。

  raw/incoming/articles/sindy-paper.md   (新規)

  Step 2: Lint で raw/incoming/ → raw//

  rw lint が frontmatter / 必須フィールドを検証。PASS なら適切な category に移動、FAIL は incoming/ に滞留。

  raw/incoming/articles/sindy-paper.md
    ↓ rw lint  → PASS
    ↓ rw ingest --only-pass
  raw/papers/local/sindy-paper.md

  Step 3: L1 → L2 への自動抽出

  rw extract-relations が LLM で entity と typed edge を抽出し、L2 Graph Ledger に append。

  .rwiki/graph/
  ├── entities.yaml         ← Brunton, SINDy 等の entity を追加
  ├── edges.jsonl           ← (Brunton)─authored→(SINDy) を candidate として append
  ├── evidence.jsonl        ← raw/papers/local/sindy-paper.md の引用箇所を登録
  └── edge_events.jsonl     ← created event を append-only で記録

  ここで重要なのは L2 は review/ 層を経由しないこと（§2.0 層別適用マトリクス、L2 は §2.12 reject-only が優先）。

  Step 4: Confidence 計算と status 自動分類

  各 candidate edge に対し confidence を計算（§2.12 の 6 係数加重式）:
  confidence = 0.35×evidence + 0.20×explicitness + 0.15×source_reliability
             + 0.15×graph_consistency + 0.10×recurrence + 0.05×human_feedback

  結果に応じて status を自動分類:

  ┌────────────┬───────────────────────┐
  │ confidence │        status         │
  ├────────────┼───────────────────────┤
  │ ≥ 0.75     │ stable（auto-accept）  │
  ├────────────┼───────────────────────┤
  │ 0.45–0.75  │ candidate（監視）       │
  ├────────────┼───────────────────────┤
  │ 0.3–0.45   │ weak（低優先）          │
  ├────────────┼───────────────────────┤
  │ < 0.3      │ reject_queue 候補      │
  └────────────┴───────────────────────┘

  Step 5: 人間の reject-only filter

  ユーザーは rw reject で「明らかに不適切な edge」だけを拒絶する。承認は不要（閾値で自動）。

  .rwiki/graph/rejected_edges.jsonl  ← reject 理由（6 分類）と pre_reject_status を記録

  Step 6: Hygiene による継続進化

  定期バッチ（default 週次）の rw graph hygiene が L2 全体に 5 ルールを適用:
  1. Decay: 未使用 edge の confidence 減衰
  2. Reinforcement: 使われた edge の強化（per-event 0.1 / per-day 0.2 上限）
  3. Competition: 同 node pair の winner 選出（L1）
  4. Contradiction tracking: 矛盾は削除せず contradiction_with: で保持
  5. Edge Merging: 類似 edge を統合

  結果として edge status が遷移し、graph が自己進化していく。

  Step 7: L3 への昇格（distill + approve）

  ここから L3 Curated Wiki への流れ:

  raw/papers/local/sindy-paper.md
    ↓ rw distill --skill paper_summary
  review/synthesis_candidates/sindy-summary.md
    ↓ 人間レビュー（review layer first 原則、§2.2）
    ↓ rw approve
  wiki/methods/sindy.md   (frontmatter `related:` は L2 から derived)

  L3 への変更は 必ず review/ 層を経由（§2.2、L3 限定原則）。承認は dangerous op category に応じて 8 段階対話 or simple confirm。

  Step 8: L3 related: の sync

  L2 edges.jsonl の変更が .rwiki/cache/stale_pages.txt に stale mark され、次の Hygiene batch で L3 frontmatter related: に一括 sync される（§Spec
  5、Hybrid stale-mark 方式）。

  # wiki/methods/sindy.md の frontmatter
  related:
    - target: wiki/methods/koopman.md
      relation: similar_approach_to
      confidence: 0.92    # L2 から derived

  ---
  Reverse 方向: 活用（query / perspective / hypothesis）

  Perspective generation (rw perspective "<topic>")

  ユーザー: "SINDy について"
    ↓
  Step 1: seed 特定（entities.yaml で正規化）
    ↓
  Step 2: L2 Graph Ledger を traverse（depth=2）
          フィルタ: status IN (stable, core) AND confidence ≥ 0.4
    ↓
  Step 3: Top M=20 件を scoring で選定
          score = 0.6×confidence + 0.3×recency + 0.1×novelty
    ↓
  Step 4: 選ばれた edges の wiki 本文を読込（L3 の信頼内容）
    ↓
  Step 5: LLM が統合分析 → 出力（stdout / review/perspectives/）
          使われた edge は usage_signal として edge_events.jsonl に append
    ↓
  [Hygiene の reinforcement 入力に]

  Hypothesis generation (rw hypothesize "<topic>")

  Perspective と類似だが、candidate edges + missing bridges から未検証命題を生成:
  score = 0.5×novelty + 0.3×confidence + 0.2×bridge_potential

  出力は必ず review/hypothesis_candidates/<topic>-<ts>.md にファイル化。

  Hypothesis verify（半自動 4 段階）

  rw verify <hypothesis-id>
    ↓
  1. LLM が raw/ から candidate evidence を 5 件抽出（grep + semantic）
  2. ユーザーが各 evidence を supporting/refuting/partial/none で評価
  3. LLM が集計判定（supporting ≥2 かつ refuting=0 → confirmed）
  4. verification_attempts 記録 + 関連 edge を reinforcement

  confirmed hypothesis は Scenario 16（query → synthesis 昇格）と同じ 8 段階対話で wiki/synthesis/ に昇格可能。

  ---
  全層を貫く原則

  Trust chain（§2.10、唯一全層適用される原則）

  任意の wiki 記述から逆方向に辿れる:
  wiki/methods/sindy.md（L3）
    └─ sources: → raw/papers/local/sindy-paper.md（L1）
    └─ related: → edge_id（L2）
                └─ evidence_ids: → evidence.jsonl の entry（L2）
                                └─ source: raw/papers/...（L1）

  履歴媒体の層別分担（§2.6）

  ┌─────┬───────────────────────────────────────────────────────┐
  │ 層  │                     補助履歴媒体                        │
  ├─────┼───────────────────────────────────────────────────────┤
  │ L1  │ Git log のみ                                           │
  ├─────┼───────────────────────────────────────────────────────┤
  │ L2  │ edge_events.jsonl（append-only event log、8 種 type）   │
  ├─────┼───────────────────────────────────────────────────────┤
  │ L3  │ frontmatter update_history:                           │
  └─────┴───────────────────────────────────────────────────────┘

  人間と LLM の役割分担（§1.3.5）

  ┌─────┬──────────────────────────────────┬─────────────────────────┐
  │ 層  │            LLM の役割             │       人間の役割          │
  ├─────┼──────────────────────────────────┼─────────────────────────┤
  │ L1  │ —                                │ 直接記述                 │
  ├─────┼──────────────────────────────────┼─────────────────────────┤
  │ L2  │ 抽出 / Hygiene / surface（主役）   │ reject only（拒絶のみ）   │
  ├─────┼──────────────────────────────────┼─────────────────────────┤
  │ L3  │ 候補生成                          │ approve 必須（従来型）     │
  └─────┴──────────────────────────────────┴─────────────────────────┘

  ---
  重要なポイント

  1. L2 は review 経由しない: 入力コスト爆発を避けるため、自動抽出 → reject-only に転換（§2.12 が §2.2 より優先）
  2. L3 は従来型 approve: trust 層として人間承認を必須に保つ
  3. Hygiene が graph を育てる: 使われる edge は強化、使われない edge は decay、自然選択的に整理
  4. Perspective / Hypothesis が L2 の最終需要者: 使用 signal が Hygiene にフィードバックされる循環

  これが Curated GraphRAG（§1.3）の核心 — 自動抽出 graph（L2）と人間キュレーション wiki（L3）の二層併走で、入力コストを削減しつつ trustを維持する設計です。

### ❯ entity と typed edgeはどうやって付与されるか

~~~
⏺ Entity と typed edge の付与経路は 5 つあります。それぞれ confidence 初期値・extraction_mode・evidence 扱いが異なります。

  概念整理

  Entity（L2 node）= 人物 / 概念 / 手法 / ツール等
    → entities.yaml に id + canonical 名 + aliases で管理
    → 例: id=brunton, canonical="Steven Brunton", aliases=["S. Brunton","Brunton"]

  Typed Edge（L2 関係）= source / type / target の三つ組
    → edges.jsonl に 1 行 1 edge で append-only
    → 例: {edge_id, source: brunton, type: authored, target: sindy-2016, confidence, status, evidence_ids, extraction_mode}

  ---
  5 経路の付与方法

  経路 1: LLM 自動抽出（rw extract-relations、主経路）

  Trigger: rw extract-relations、または Maintenance autonomous の推奨（typed-edge 整備率 < 2.0 で発火）

  Mechanism: LLM が 2-stage で raw/ や wiki/ のテキストから抽出
  1. Entity 抽出: 文章から entity を識別 → entities.yaml に追加（既存 alias と照合、normalize）
  2. Relation 抽出: 文章から関係を識別 → edges.jsonl に candidate として append

  Extraction mode の決定:

  ┌───────────────┬─────────────────┬────────────────────────────────────────────────────────────────┐
  │    モード     │ confidence 係数 │                            判定基準                            │
  ├───────────────┼─────────────────┼────────────────────────────────────────────────────────────────┤
  │ explicit      │       1.0       │ 「A は B を著した」のように直接的記述                          │
  ├───────────────┼─────────────────┼────────────────────────────────────────────────────────────────┤
  │ paraphrase    │      0.75       │ 「A の手法は B」のように言い換えで含意                         │
  ├───────────────┼─────────────────┼────────────────────────────────────────────────────────────────┤
  │ inferred      │       0.5       │ 文脈推論（「A グループは B 系を研究」→ A が B を著した蓋然性） │
  ├───────────────┼─────────────────┼────────────────────────────────────────────────────────────────┤
  │ co_occurrence │      0.25       │ 同一段落内で共起するが関係明示なし                             │
  └───────────────┴─────────────────┴────────────────────────────────────────────────────────────────┘

  Confidence 初期値: 6 係数加重式（§2.12）で計算
  confidence = 0.35×evidence + 0.20×explicitness + 0.15×source_reliability
             + 0.15×graph_consistency + 0.10×recurrence + 0.05×human_feedback

  例: explicit で source が論文（高信頼）→ explicitness=1.0、source_reliability=0.9 → 初期 confidence ≈ 0.65（candidate status）

  Evidence: 抽出元の raw ファイル＋引用箇所（quote / span）が evidence.jsonl に登録、edge は evidence_ids: [ev_xxx] で参照

  出力先: review/relation_candidates/（任意レビュー）→ 直接 edges.jsonl に append（reject-only filter）

  ---
  経路 2: Frontmatter shortcut の正規化（normalize_frontmatter API）

  Trigger: rw ingest / rw approve / rw graph rebuild 実行時に自動 invoke（決定 5-1）

  Mechanism: L3 wiki ページの frontmatter にある entity 固有 shortcut field を typed edge に展開

  例: wiki/entities/people/brunton.md の frontmatter
  authored: [sindy-2016, e-sindy-2018]
  collaborated_with: [kutz, rudy]
  mentored: [champion]

  これを Spec 5 の normalize_frontmatter() API が以下のロジックで展開:
  1. relations.yml の mapping table を参照（authored の inverse は authored_by、symmetric は false）
  2. source = brunton（当該ページ）、target = field の値
  3. extraction_mode = explicit（人間明示記述、最高 explicitness）
  4. inverse: / symmetric: に従い双方向 edge を自動生成:
    - (brunton)─authored→(sindy-2016) + (sindy-2016)─authored_by→(brunton)
  5. evidence_ids = 空（frontmatter 自体が evidence、evidence.jsonl に特殊 source "frontmatter" で登録）
  6. confidence = 0.9 固定（人間が直接記述した root of trust として高値）

  冪等性: 同一ページを複数回 normalize しても duplicate を作らない（edge_id = hash(source + type + target) で upsert）

  Alias 衝突時: relations.yml の canonical: 値を優先、曖昧時は警告出力 → user に手動 resolve を要求

  ---
  経路 3: [[link]] syntax の untyped edge → 後付け typing

  Trigger: wiki 本文中の [[wiki/methods/koopman]] 等の Obsidian 風 link

  Mechanism:
  1. ingest / rebuild 時に [[link]] を untyped edge として登録
  {"edge_id": "...", "source": "sindy", "type": "untyped", "target": "koopman", "confidence": 0.4, "status": "weak", "extraction_mode":
  "co_occurrence"}
  2. 後で rw extract-relations か rw audit graph --propose-relations が untyped を semantic 解析して typed 化を提案
    - 例: untyped(sindy → koopman) → 提案: similar_approach_to
  3. 提案は review/relation_candidates/ 経由でレビュー、ユーザー承認で typed に昇格

  意義: 人間が wiki を書く時の自然な link 構文を活用しつつ、後から graph に取り込める段階的 typing 戦略

  ---
  経路 4: Hypothesis verify による reinforcement / 新規生成

  Trigger: rw verify <hypothesis-id> で hypothesis が confirmed される時

  Mechanism:
  - Hypothesis が「A と B の relation R が成立する」という命題を含む場合
  - confirmed 判定時に対応 edge をチェック:
    - 既存なら reinforcement（confidence delta、per-event cap 0.1）
    - 不在なら 新規 candidate edge を作成（extraction_mode=inferred、evidence は verification 時の supporting evidence_ids）
  - edge_events.jsonl に reinforced(Direct) event を追記

  意義: 仮説検証のフィードバックループが graph を育てる（§1.3.5 哲学「使われる知識を残す」、Hygiene の Reinforcement ルール）

  ---
  経路 5: 手動 CRUD（P0、動作確認用）

  Trigger: rw edge create / promote / demote 等（Spec 5 P0 で実装、Simple dangerous op）

  Mechanism:
  - ユーザーが直接 edges.jsonl に edge を追加する手段
  - 主に動作確認 / migration / 例外的な手動修正用
  - 理由記録必須（§2.5 Simple dangerous op、edge_events.jsonl の reason field）
  - extraction_mode は explicit、confidence は user 指定（default 0.7）

  注意: 通常運用では使わない。LLM 抽出 + Hygiene が主軸であり、手動 CRUD は escape hatch

  ---
  経路別の特徴比較

  ┌───────────────────────┬───────────────────┬──────────────────────────────────────────┬─────────────────────────────┬────────────────────────┐
  │         経路          │      主動者       │             extraction_mode              │       初期 confidence       │        Evidence        │
  ├───────────────────────┼───────────────────┼──────────────────────────────────────────┼─────────────────────────────┼────────────────────────┤
  │ 1. LLM 抽出           │ LLM               │ explicit / paraphrase / inferred /       │ 6 係数式で計算（0.3-0.85）  │ raw 引用               │
  │                       │                   │ co_occurrence                            │                             │                        │
  ├───────────────────────┼───────────────────┼──────────────────────────────────────────┼─────────────────────────────┼────────────────────────┤
  │ 2. frontmatter        │ Spec 5 API        │ explicit                                 │ 0.9 固定                    │ "frontmatter" 特殊     │
  │ normalize             │                   │                                          │                             │ source                 │
  ├───────────────────────┼───────────────────┼──────────────────────────────────────────┼─────────────────────────────┼────────────────────────┤
  │ 3. [[link]] typing    │ 自動 +            │ co_occurrence → 後付け                   │ 0.4 → typed 昇格時に再計算  │ 出現箇所               │
  │                       │ ユーザー承認      │                                          │                             │                        │
  ├───────────────────────┼───────────────────┼──────────────────────────────────────────┼─────────────────────────────┼────────────────────────┤
  │ 4. Hypothesis verify  │ LLM + ユーザー    │ inferred                                 │ reinforcement（既存）or     │ verification evidence  │
  │                       │                   │                                          │ 新規                        │                        │
  ├───────────────────────┼───────────────────┼──────────────────────────────────────────┼─────────────────────────────┼────────────────────────┤
  │ 5. 手動 CRUD          │ ユーザー          │ explicit                                 │ user 指定                   │ reason record          │
  └───────────────────────┴───────────────────┴──────────────────────────────────────────┴─────────────────────────────┴────────────────────────┘

  ---
  全経路で共通する原則

  1. Evidence 必須（§2.12 ルール 1）: evidence なしの edge は confidence ≤ 0.3 にクランプ。経路 5 だけ「reason record」が evidence の代用
  2. Append-only: edges.jsonl は upsert（既存は edge_id で更新）、edge_events.jsonl は純粋 append
  3. 冪等性: 同じ source/type/target は edge_id が同じ hash になり、複数経路から付与されても重複しない
  4. Hygiene で育つ: どの経路で生まれた edge も、その後は同じ Hygiene サイクル（Decay / Reinforcement / Competition / Contradiction tracking / Edge
  Merging）に乗る

  ---
  典型的なライフサイクル

  [経路 1: LLM 抽出]
    └─ candidate (confidence 0.55)

  [経路 4: Hypothesis verify で reinforce]
    └─ candidate → confidence 0.65

  [Hygiene: Decay 1 週間 → Reinforcement Direct usage 3 回]
    └─ candidate → confidence 0.78 → status: stable に auto-promote (≥0.75)

  [Perspective で 10 回使用 + 評価高]
    └─ stable → confidence 0.92 → status: core に手動 promote

  このように entity / edge は 多経路で生まれ、Hygiene で育ち、使用で強化される 動的な存在として L2 Graph Ledger に蓄積されていきます。

  Relation type は 3 階層で管理されます。

  ---
  relations.yml の管理スキーマ

  各 relation type は .rwiki/vocabulary/relations.yml で 4 属性を持ちます:

  relations:
    authored:
      description: "Person が Paper / Method を執筆・創出"
      inverse: authored_by
      symmetric: false
      domain: [person]            # source の Entity type 制約
      range: [method, paper]      # target の Entity type 制約

    similar_approach_to:
      description: "同じ原理の別実装"
      inverse: similar_approach_to   # self-inverse
      symmetric: true
      domain: [method, concept]
      range: [method, concept]

    uses:
      description: "A が B を利用する"
      inverse: used_by
      symmetric: false
      domain: [method, tool, project]
      range: [method, tool, concept]

  属性の役割:
  - inverse: 双方向 edge 生成のペア（self なら symmetric、または unidirectional）
  - symmetric: true なら 1 方向の入力で双方向 edge が等価
  - domain / range: Entity type 制約。audit graph がこれを違反する edge を flag（例: concept → person の authored は無効）

  ---
  拡張方針

  最初から大きな ontology を作らない（Spec 5 Key Req 5）:
  1. MVP 12 + 抽象 8 + Entity 固有 10 ≈ 30 種で開始
  2. 使用実態から必要な relation を順次追加
  3. rw audit graph --propose-relations が「同じ意味の異なる relation type が混在」を検出して整理を促す
  4. Phase 3 以降の Edge Merging（Hygiene ルール 5）が類似 relation を自動統合候補化

  追加の判断基準:
  - 新 relation で表現したい意図が既存で表現できるか
  - 既存 relation の semantic 変化を引き起こさないか
  - LLM 抽出 prompt に新 relation を組み込んだ時の精度

### lightweight ontology

~~~
❯ この部分がgraphRAGのRDFに相当する部分で、オントロジーの役割を与えているという理解で良いか

⏺ ほぼその理解で正しいですが、いくつか重要な nuance があります。整理します。

  結論

  ┌──────────────────────────────┬────────────────────────────────────────────────────────┐
  │             観点             │                         一致度                         │
  ├──────────────────────────────┼────────────────────────────────────────────────────────┤
  │ RDF triple 構造との同型性    │ ✅ 高（subject/predicate/object ≈ source/type/target） │
  ├──────────────────────────────┼────────────────────────────────────────────────────────┤
  │ Lightweight ontology の役割  │                         ✅ Yes                         │
  ├──────────────────────────────┼────────────────────────────────────────────────────────┤
  │ 形式的 RDF/OWL 準拠          │         ❌ No（独自 JSONL、推論エンジンなし）          │
  ├──────────────────────────────┼────────────────────────────────────────────────────────┤
  │ 進化する ontology という性質 │                     ✅ Rwiki 独自                      │
  └──────────────────────────────┴────────────────────────────────────────────────────────┘

  ---
  1. RDF との対応関係（高い同型性）

  ┌────────────────────────────────────────┬───────────────────────────────────────────────────────────┐
  │            RDF / RDFS / OWL            │                   Rwiki L2 Graph Ledger                   │
  ├────────────────────────────────────────┼───────────────────────────────────────────────────────────┤
  │ Triple: <subject> <predicate> <object> │ Edge: {source, type, target}（edges.jsonl）               │
  ├────────────────────────────────────────┼───────────────────────────────────────────────────────────┤
  │ IRI / URI                              │ entity_id（entities.yaml の id）                          │
  ├────────────────────────────────────────┼───────────────────────────────────────────────────────────┤
  │ Class（rdfs:Class）                    │ Entity の type: field（person / concept / method / tool） │
  ├────────────────────────────────────────┼───────────────────────────────────────────────────────────┤
  │ Property（rdf:Property）               │ Relation type（relations.yml の各 entry）                 │
  ├────────────────────────────────────────┼───────────────────────────────────────────────────────────┤
  │ rdfs:domain / rdfs:range               │ そのまま domain: / range: で同名同義                      │
  ├────────────────────────────────────────┼───────────────────────────────────────────────────────────┤
  │ owl:inverseOf                          │ そのまま inverse: で同名同義                              │
  ├────────────────────────────────────────┼───────────────────────────────────────────────────────────┤
  │ owl:SymmetricProperty                  │ symmetric: true                                           │
  ├────────────────────────────────────────┼───────────────────────────────────────────────────────────┤
  │ rdfs:label / skos:altLabel             │ canonical: / aliases:                                     │
  └────────────────────────────────────────┴───────────────────────────────────────────────────────────┘

  つまり Rwiki は意図的に RDFS の語彙概念を採用しています。relations.yml の構造は RDF Schema を読み下した人なら即座に理解できる設計です。

  ---
  2. Ontology としての役割（lightweight）

  オントロジーが提供する 4 機能と Rwiki の対応:

  ┌────────────────────────────────────┬─────────────────────────────────────────────────────────────────────────────────────────────────────┐
  │           Ontology 機能            │                                           Rwiki での実装                                            │
  ├────────────────────────────────────┼─────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ 語彙統制（vocabulary control）     │ ✅ relations.yml で許可される type を列挙、entities.yaml で alias 統合                              │
  ├────────────────────────────────────┼─────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ 型階層（taxonomy）                 │ ⚠ 浅い（person / concept / method / tool の 1 階層のみ、サブクラス階層なし）                        │
  ├────────────────────────────────────┼─────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ 関係性制約（property constraints） │ ✅ domain / range が制約として機能、audit graph が違反検出                                          │
  ├────────────────────────────────────┼─────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ 推論（inferencing）                │ ❌ 形式的推論エンジンなし。Perspective / Hypothesis 生成は LLM ベースで、論理的 entailment ではない │
  └────────────────────────────────────┴─────────────────────────────────────────────────────────────────────────────────────────────────────┘

  結論: lightweight ontology としては機能しているが、Description Logic（OWL）レベルの厳密性は持たない。これは意図的な設計選択です。

  ---
  3. 通常の GraphRAG との位置関係

  ここが重要な区別:

  ┌───────────┬──────────────────────────────────────────────────┬───────────────────────────┬─────────────────────────────────────────────────┐
  │           │    通常の GraphRAG（MS GraphRAG / LightRAG）     │       RDF/OWL 系 KG       │             Rwiki Curated GraphRAG              │
  ├───────────┼──────────────────────────────────────────────────┼───────────────────────────┼─────────────────────────────────────────────────┤
  │ Schema    │ Schemaless（自由抽出）                           │ 厳密 schema 必須          │ 半 schema-driven（relations.yml                 │
  │           │                                                  │                           │ で語彙制約、拡張可）                            │
  ├───────────┼──────────────────────────────────────────────────┼───────────────────────────┼─────────────────────────────────────────────────┤
  │ Triple    │ 自由形式（{head, relation, tail} だが relation   │ 厳密 IRI 三つ組           │ 構造は RDF 同型、ID は独自                      │
  │ 形式      │ は文字列）                                       │                           │                                                 │
  ├───────────┼──────────────────────────────────────────────────┼───────────────────────────┼─────────────────────────────────────────────────┤
  │ 推論      │ embedding ベース類似性                           │ 論理的推論（subClassOf    │ LLM + Hygiene による進化的整理（推論ではない）  │
  │           │                                                  │ 等）                      │                                                 │
  ├───────────┼──────────────────────────────────────────────────┼───────────────────────────┼─────────────────────────────────────────────────┤
  │ 進化      │ 静的（再構築が前提）                             │ 静的（schema は固定）     │ 動的（Hygiene で confidence 進化）              │
  └───────────┴──────────────────────────────────────────────────┴───────────────────────────┴─────────────────────────────────────────────────┘

  つまり Rwiki は 「RDF と GraphRAG の中間」 に位置します:
  - RDF より自由（schema 拡張可、推論不要）
  - GraphRAG よりも構造的（vocabulary 統制、domain/range 制約）

  ---
  4. Rwiki が RDF に追加した独自の概念

  伝統的 RDF にない、Rwiki 固有の属性:

  {
    "edge_id": "e_042",
    "source": "brunton",
    "type": "authored",
    "target": "sindy-2016",

    // ↑ ここまでは RDF triple と同等
    // ↓ ここから Rwiki 独自（伝統的 RDF にはない）

    "confidence": 0.92,           // 確信度（LLM 抽出由来 or human）
    "status": "stable",           // edge lifecycle（weak/candidate/stable/core/deprecated/rejected）
    "evidence_ids": ["ev_034"],   // 証拠への逆参照
    "extraction_mode": "explicit",// 抽出方法
    "created_at": "2026-04-25T...",
    "updated_at": "2026-04-25T..."
  }

  これらは形式的には RDF reified statement（statement について statement する仕組み）で表現可能ですが、実用上は重すぎるため Rwiki は edge を
  first-class object にして直接属性を持たせる設計を選択。

  さらに以下が Rwiki 独自:
  - edge_events.jsonl: edge 状態の時系列ログ（伝統的 KG にはない時間軸）
  - Hygiene 5 ルール: ontology 自体が時間で進化する（伝統的 ontology は人間が手動編集）
  - Reject-only filter: ontology 構築の労力を最小化（§2.12 の核心）

  ---
  5. つまりこういうこと

  「Curated Knowledge Graph に対する探索・発見システム」（§1.3.1 ワンライン定義）の中の Knowledge Graph 部分 が、RDF + lightweight ontology
  に相当します。具体的には:

  RDF triple 構造 ──────────── ✓ そのまま採用
     ↓
  RDFS vocabulary（domain/range/inverse） ─────── ✓ そのまま採用
     ↓
  OWL 推論エンジン ─────────── ✗ 採用しない（LLM 統合分析で代替）
     ↓
  + evidence-backed（confidence + evidence_ids） ─── ✓ Rwiki 独自
  + evolving（Hygiene 5 ルール） ───────────── ✓ Rwiki 独自
  + append-only ledger（git 管理） ─────────── ✓ Rwiki 独自

  設計哲学:
  - 「形式的厳密性」より「実用的進化性」を優先
  - ontology 構築コストを Hygiene + LLM で自動化
  - ただし語彙統制（relations.yml）は維持して**「LLM の妄想ネットワーク化」を防ぐ**

  このバランスが §1.3 で言う Curated GraphRAG の核心、つまり「ゆるい RDF + Hygiene による生きた ontology」という位置付けです。

  ご理解の通り 「GraphRAG の RDF 相当 + ontology の役割」 を持っていますが、伝統的 RDF/OWL ほど厳密ではなく、進化性を備えた実用志向の lightweightontology だと言えます。
