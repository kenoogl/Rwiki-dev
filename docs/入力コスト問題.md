# 1. relation抽出の基本方針

Rwikiでは relation を最初から「正解」として扱わない方がよいです。

## relation = 仮説

として扱う。

つまり：

```text
A relates_to B
```

ではなく、

```text
A may_relates_to B
confidence: 0.72
evidence: source.md の該当箇所
status: candidate
```

にする。

これが重要です。

------

# 2. 抽出対象は3層に分ける

いきなり複雑な関係を全部取ると壊れます。

## Layer 1：明示的relation

本文に直接書かれている関係。

例：

```text
AはBの原因である
AはBを改善する
AはBを用いる
AはBに依存する
```

これは比較的安全。

------

## Layer 2：近接・共起relation

同じ段落・同じ実験・同じ議論に出たもの。

例：

```text
A and B appear in same context
A is discussed_with B
```

これは弱い関係。

------

## Layer 3：推論relation

LLMが意味的に推定する関係。

例：

```text
A may_explain B
A is_alternative_to B
A is_tradeoff_with B
A is_generalization_of B
```

これは便利だが危険。

------

# 3. 最初に使うべきrelation type

最小実装では、relation typeを増やしすぎない方がよいです。

おすすめはこの10個程度です。

```yaml
relation_types:
  - uses
  - depends_on
  - causes
  - improves
  - degrades
  - compares_with
  - alternative_to
  - part_of
  - supports
  - contradicts
```

加えて、弱い関係として：

```yaml
  - co_occurs_with
  - related_to
```

を置く。

ポイントは、**最初から立派なontologyを作らない**ことです。

------

# 4. 抽出アルゴリズムの流れ

## Step 1：Entity抽出

まず本文から重要概念を取る。

```text
入力文書
↓
重要Entity抽出
↓
正規化
↓
Entity候補リスト
```

例：

```yaml
entities:
  - Rwiki
  - GraphRAG
  - relation extraction
  - approve flow
  - input cost
```

------

## Step 2：Relation候補抽出

Entityペアに対して関係を抽出する。

```yaml
relations:
  - source: Rwiki
    relation: has_problem
    target: input cost
    confidence: 0.92
    evidence: "Rwikiは...人間がかなり手を入れる必要がある"
```

ただし `has_problem` のような自由relationを許すと増殖するので、内部では既定typeに正規化する。

```yaml
relation: suffers_from
```

ではなく、

```yaml
relation: depends_on
# または
relation: degrades
```

のように寄せる。

------

## Step 3：Evidence必須化

relationには必ず根拠を付ける。

```yaml
evidence:
  file: raw/logs/xxx.md
  quote: "GraphRAG以前にGraphが育たない"
  span: line 120-124
```

これがない relation は採用しない。

------

## Step 4：Confidence付与

confidenceは単なるLLMの自己評価では弱いので、複数要素で決める。

```text
confidence =
  明示度
+ evidenceの強さ
+ 既存graphとの整合性
+ 複数sourceでの再出現
- contradiction
```

初期実装では単純でよいです。

```yaml
confidence:
  explicit: 0.9
  co_occurrence: 0.4
  inferred: 0.6
```

------

# 5. 最小実装の出力形式

Rwikiではまずこれくらいで十分です。

relation typeは後から増やす。配置も適切なところでOK.核心はconfidenceをどう計算するか



核心は、**confidenceを「LLMの自信」ではなく「証拠の強さ」として計算する**ことです。

私ならこうします。

# Confidence = relationの採用優先度

まず confidence は「真偽確率」ではなく、

> この relation をGraphに入れて使ってよい度合い

と定義します。

------

# 1. 基本式

```text
confidence =
  0.35 * evidence_score
+ 0.20 * explicitness_score
+ 0.15 * source_reliability
+ 0.15 * graph_consistency
+ 0.10 * recurrence_score
+ 0.05 * human_feedback
```

初期実装ではこれで十分です。

------

# 2. 各スコアの意味

## evidence_score

根拠引用がどれだけ直接的か。

```text
0.9〜1.0 明示的に書いてある
0.6〜0.8 文脈からかなり自然
0.3〜0.5 共起しているだけ
0.0〜0.2 根拠が弱い
```

最重要です。

------

## explicitness_score

関係語が本文にあるか。

例：

```text
AはBを引き起こす
AはBに依存する
AはBと比較される
```

なら高い。

```text
同じ段落にAとBがある
```

なら低い。

------

## source_reliability

入力元の信頼度。

```text
reviewed paper        0.9
approved synthesis    0.9
user-authored note    0.8
discussion log        0.6
auto summary          0.5
web clip              0.4〜0.7
```

Rwikiでは、**raw log由来relationは低め、synthesis由来relationは高め**がよいです。

------

## graph_consistency

既存Graphと矛盾しないか。

```text
既存edgeと補強関係      +高
同じクラスタ内          +中
既存edgeと矛盾          -大
孤立                   +低
```

ただし、矛盾を即削除しない。
`contradicts` として保持する。

------

## recurrence_score

複数sourceで同じ関係が出たか。

```text
1 source     0.2
2 sources    0.5
3+ sources   0.8
独立source多数 1.0
```

重要なのは、**同じ会話ログ内の繰り返しを過大評価しない**ことです。

------

## human_feedback

人間が明示的に触ったか。

```text
rejected         0
edited           0.7
accepted         1.0
used_in_answer   +0.1〜0.2
```

------

# ■ 2. 役割の本質

Graph Hygieneの役割はこれです：

> **「正しさ」ではなく「生き残りやすさ」を決める**

つまり：

- 正しいが使われない → 消える
- 少し間違ってても使われる → 残る

👉 かなり重要な思想転換です

------

# ■ 3. 3つの基本ダイナミクス

Graph Hygieneは最低限この3つで構成します。

------

## ① Decay（忘却）

使われないrelationは弱くなる。

```text
confidence(t+1) = confidence(t) * (1 - λ)
```

例：

```text
λ = 0.01 / day
```

👉 「放置＝削除」に近づく

------

## ② Reinforcement（強化）

使われたrelationは強くなる。

```text
confidence += α * usage_signal
```

usage_signal例：

- queryで使用
- explanationで採用
- 他relationの推論に使われた

👉 **queryが学習になる**

------

## ③ Competition（競争）

似たrelation同士が競う。

例：

```text
A → B (causes)
A → B (correlates_with)
```

→ どちらかが優勢になる

```text
confidence_i = confidence_i / Σ(similar_relations)
```

👉 Graphの冗長性が減る

------

# ■ 4. さらに重要な2つ（ここが肝）

------

## ④ Contradiction tracking（矛盾管理）

矛盾は削除しない。

```yaml
A causes B
A does_not_cause B
```

↓

```yaml
relation:
  type: contradicts
```

そして：

```text
両方のconfidenceを少し下げる
```

👉 真理判定ではなく「未確定状態」を保持

------

## ⑤ Merging（同一化）

似たrelationを統合する。

```text
improves ≒ enhances ≒ increases
```

↓

```text
improves に吸収
```

👉 ontologyを後から自然形成

------

# ■ 5. 実際の更新式（シンプル版）

1ステップでやるなら：

```text
confidence_next =
  confidence_current
  + α * usage
  + β * recurrence
  - γ * decay
  - δ * contradiction
```

初期値：

```text
α = 0.05
β = 0.03
γ = 0.01
δ = 0.1
```

------

# ■ 6. 状態遷移

relationは状態を持たせるとよいです。

```yaml

  - weak
  - candidate
  - stable
  - core
  - deprecated
```

```text

stable → (頻用) → core
candidate → (使われない) → weak
weak → (さらに放置) → deprecated
```

👉 Graphが「呼吸」する

どっち行きます？
ここから一気に“動く設計”に入れます。



ここはかなり重要で、設計を一歩間違えると
**「人気＝真実」になってGraphが壊れます。**

結論から言うと：

> **usage_signalは「使われた回数」ではなく
> 「推論にどれだけ貢献したか」で定義するべきです。**

------

# ■ 1. usage_signalの基本定義

まずシンプルに書くと：

```text
usage_signal = 貢献度 × 信頼度 × 独立性
```

------

# ■ 2. 「使われた」の種類を分解する

usageは1種類ではダメです。最低でも4種類に分ける。

------

## ① Direct Usage（直接使用）

relationが回答生成に直接使われた

例：

```text
Q: Rwikiの問題は？
A: 入力コスト
```

内部：

```text
Rwiki → has_problem → input_cost
```

👉 最も強い

```text
score = 1.0
```

------

## ② Support Usage（補助的使用）

論理展開の一部で使われた

例：

```text
input_cost → degrades → graph_growth
```

👉 中程度

```text
score = 0.6
```

------

## ③ Retrieval Usage（検索ヒット）

検索にはヒットしたが、回答には使われなかった

👉 弱い

```text
score = 0.2
```

------

## ④ Co-activation（同時活性）

同じqueryで一緒に呼ばれた

👉 ノイズ多め

```text
score = 0.1
```

------

# ■ 3. 重要：貢献度の重み付け

usage_signalの核心はここです。

------

## 貢献度 = reasoningでの役割

```text
usage_contribution =
  answer_core_weight * is_core
+ chain_weight * position_in_chain
```

例：

```text
最終結論に直結 → 1.0
中間推論       → 0.6
周辺情報       → 0.3
```

------

# ■ 4. 信頼度補正

低confidenceのrelationが使われても強化しすぎない。

```text
usage_signal *= sqrt(confidence)
```

👉 低品質relationの暴走防止

------

# ■ 5. 独立性（超重要）

これを入れないと壊れます。

------

## 問題

同じソースから派生したrelationが何度も使われる：

👉 偽の強化

------

## 解決

```text
independence_factor =
  1 / (同一sourceからの使用回数)
```

または：

```text
distinct_sources_count を使う
```

------

# ■ 6. 時間減衰（Temporal weighting）

最近のusageを優先：

```text
time_weight = exp(-Δt / τ)
```

例：

```text
τ = 7日
```

------

# ■ 7. 最終式

まとめると：

```text
usage_signal =
  base_usage_score
  × contribution_weight
  × sqrt(confidence)
  × independence_factor
  × time_weight
```

------

# ■ 8. 例

```yaml
relation: Rwiki → has_problem → input_cost
```

### ケース1：コア回答で使用

```text
base_usage_score = 1.0
contribution = 1.0
confidence = 0.8 → sqrt = 0.89
independence = 1.0
time_weight = 1.0

usage_signal ≈ 0.89
```

------

### ケース2：補助的に使用

```text
base = 0.6
contribution = 0.6
confidence = 0.7 → 0.84
independence = 0.5
time_weight = 0.9

usage_signal ≈ 0.14
```

------

# ■ 9. どう記録するか（実装）

relationごとにこれを持つ：

```yaml
usage:
  total_score: 3.42
  last_used: 2026-04-24
  use_count: 7
  distinct_sources: 3
  contexts:
    - query_id: q_001
      contribution: core
      score: 0.89
```

------

# ■ 10. 最重要ポイント

usage_signal設計で一番やってはいけないこと：

------

## ❌ NG

```text
使われた回数 += 1
```

------

## ✅ 正解

```text
推論への寄与を加算
```

------

# ■ 11. Rwiki的に重要な補正

Rwikiではさらにこれを入れると強い：

------

## ▶ Evidence再利用ボーナス

同じevidenceが複数回使われる：

```text
usage_signal += 0.1
```

👉 「引用される事実」が強化される

------

## ▶ Cross-contextボーナス

異なる文脈で使われる：

```text
usage_signal += 0.2
```

👉 汎用知識が強くなる

------

# ■ 12. 本質

usage_signalは：

```text
「このrelationがどれだけ思考に使われたか」
```

であって、

```text
「どれだけ頻出したか」
```

ではない。

------

# ■ 13. 次に来る核心

ここまで来ると、次はこれです：

👉 **competitionをどの粒度でやるか**

- 同一nodeペアだけ？
- relation typeも含める？
- embedding距離でやる？

ここでGraphの形が決まります。

行きます？



では **competitionの粒度**です。
結論から言うと、最初は **3段階 competition** がよいです。

# 1. Level 1：同一ノードペア内 competition

最小・必須です。

```text
A → B
```

に対して複数relationがある場合：

```text
A causes B
A supports B
A correlates_with B
A discusses B
```

これらを競わせる。

目的は、**同じA-B間で関係ラベルが増殖するのを防ぐ**こと。

------

# 2. Level 2：類似ノードペア competition

次に必要です。

例：

```text
Rwiki → input cost
Rwiki system → input burden
Rwiki workflow → curation cost
```

これらは実質的に近い。

ここではノード正規化・merge候補を出す。

目的は、**同じ概念が別名で増殖するのを防ぐ**こと。

------

# 3. Level 3：意味的に競合するrelation competition

これは高度です。

例：

```text
manual curation improves graph quality
manual curation degrades scalability
```

これは矛盾ではなく、**トレードオフ**です。

この場合、片方を潰さず、

```text
manual curation --improves--> graph quality
manual curation --degrades--> scalability
manual curation --has_tradeoff--> quality_vs_scalability
```

のように整理する。

目的は、**矛盾・対立・トレードオフを知識として残す**こと。

------

# 推奨実装順

最初はこれで十分です。

```text
Phase 1:
  同一node pairだけcompetition

Phase 2:
  類似node merge候補

Phase 3:
  tradeoff / contradiction整理
```

------

# competition score

各relationに勝敗をつけるなら：

```text
competition_score =
  confidence
+ usage_strength
+ recurrence
+ human_feedback
- redundancy_penalty
```

ただし、削除ではなく状態変更にします。

```text
winner      → stable
runner-up   → candidate
loser       → weak
obsolete    → deprecated
```

------

# 重要な設計原則

**competitionは「正解を1つに決める処理」ではありません。**

Rwikiではこう捉えるべきです。

```text
competition = graphを圧縮・整理する処理
```

つまり：

- 似たものをまとめる
- 弱いものを下げる
- 矛盾を明示する
- tradeoffを発見する

------

# 最小仕様に落とすなら

```yaml
competition:
  scope:
    - same_node_pair
    - similar_node_pair_later
    - semantic_conflict_later

  action:
    - promote
    - demote
    - merge_candidate
    - mark_tradeoff
    - mark_contradiction

  never:
    - hard_delete_without_log
```

これで、Graphが汚れながらも自浄作用を持ちます。



適切なのは、最初から本格Graph DBにせず、

**Git管理できるファイルGraphを正本にし、必要に応じてSQLite/NetworkX等で派生インデックスを作る方式**です。

# 結論

Rwikiではこうするのがよいです。

```text
Markdown / YAML / JSONL = 正本
SQLite / NetworkX / Graph DB = 派生キャッシュ
```

つまり、Graph DBを正本にしない。

------

# 1. なぜGraph DB正本にしないか

Neo4jなどを最初から正本にすると：

- Gitで差分管理しにくい
- relationの発生根拠を追いにくい
- 人間がレビューしにくい
- Rwikiの「軽量・可搬」思想とズレる
- confidence更新履歴がブラックボックス化しやすい

ので、初期段階では重いです。

------

# 2. 推奨ディレクトリ構成

```text
wiki/
  graph/
    entities.yaml
    edges.jsonl
    edge_events.jsonl
    rejected_edges.jsonl
    merge_candidates.yaml
    graph_stats.yaml
```

特に重要なのは `edges.jsonl` と `edge_events.jsonl` です。

------

# 3. edgeはJSONLがよい

YAMLは人間には読みやすいですが、大量relationには向きません。
edge本体はJSONLがよいです。

```json
{"edge_id":"e_001","source":"Rwiki","type":"depends_on","target":"auto_relation_extraction","confidence":0.82,"status":"stable","evidence_ids":["ev_001"],"created_at":"2026-04-24","updated_at":"2026-04-24"}
```

1行1edgeにすると：

- appendしやすい
- diffしやすい
- スクリプト処理しやすい
- 壊れにくい

------

# 4. confidence更新はevent logで管理

confidenceを直接上書きするだけだと、なぜ上がったか分からなくなります。

なので：

```text
edges.jsonl        = 現在状態
edge_events.jsonl  = 更新履歴
```

に分ける。

例：

```json
{"edge_id":"e_001","event":"used_in_query","delta":0.04,"query_id":"q_023","reason":"core_answer","timestamp":"2026-04-24"}
{"edge_id":"e_001","event":"decay","delta":-0.01,"timestamp":"2026-04-25"}
{"edge_id":"e_001","event":"human_reject","delta":-1.0,"timestamp":"2026-04-26"}
```

これはかなり重要です。

------

# 5. Evidenceはedgeから分離する

relation本体に長い引用を埋め込むと重くなります。

```text
evidence.jsonl
```

を別に持つのがよいです。

```json
{"evidence_id":"ev_001","file":"raw/logs/2026-04-24.md","quote":"relation抽出はデフォルトONに近いUX","span":"L20-L24"}
```

edge側は evidence_id だけ参照します。

------

# 6. 実装モデル

最小構成はこれです。

```text
正本:
  entities.yaml
  edges.jsonl
  evidence.jsonl
  edge_events.jsonl

処理:
  rw graph build
  rw graph score
  rw graph hygiene
  rw graph query

派生:
  .rw/cache/graph.sqlite
  .rw/cache/networkx.pkl
```

------

# 7. SQLiteをキャッシュに使う

検索やquery時にはJSONLを毎回読むと遅いので、SQLiteにビルドします。

```text
edges.jsonl → graph.sqlite
```

SQLiteには：

```text
nodes
edges
evidence
events
```

を入れる。

ただし、SQLiteは正本ではなくキャッシュ。

------

# 8. query時の流れ

```text
user query
  ↓
vector検索
  ↓
関連entity抽出
  ↓
graph.sqliteで近傍探索
  ↓
confidence/statusでfilter
  ↓
answer生成
  ↓
used edgeをedge_events.jsonlに追記
  ↓
次回hygieneでconfidence更新
```

------

# 9. status設計

edgeには状態を持たせます。

```text
weak
candidate
stable
core
deprecated
rejected
```

queryで使うのは基本：

```text
stable + core + candidate上位
```

`weak` は探索用、`deprecated/rejected` は通常使わない。

------

# 10. 最初の実装方針

最初はこれで十分です。

```text
Phase 0:
  edges.jsonl + evidence.jsonl + simple scorer

Phase 1:
  SQLite cache + query近傍探索

Phase 2:
  usage event + confidence更新

Phase 3:
  competition / merge / contradiction

Phase 4:
  必要ならNeo4j等へexport
```

------

# 一番大事な設計判断

RwikiではGraphを、

```text
巨大なGraph DB
```

ではなく、

```text
証拠付きedgeの台帳
```

として実装するのがよいです。

これなら、Rwikiの思想である

- Gitで管理できる
- 人間が読める
- 差分が追える
- 根拠に戻れる
- LLMが更新できる

を保ったまま、GraphRAG的に育てられます。
