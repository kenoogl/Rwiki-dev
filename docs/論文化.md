❯ 論文化するとしたら、新規性の点を挙げて

⏺ 論文化を想定した新規性ポイントを、既存研究との差分で整理します。

  主要 novelty 候補（強度順）

  ★★★ 1. Hygiene as Explicit KG Evolution Semantics

  主張: Knowledge Graph に 5 つの operational evolution rule（Decay / Reinforcement / Competition / Contradiction tracking / Edge Merging）を明示的に定義し、confidence の deterministic 進化方程式として定式化:

  confidence_{t+1} = confidence_t + α × usage_signal + β × recurrence
                                    − γ × decay − δ × contradiction

  既存研究との差分:
  - KG completion / link prediction（Bordes 2013、TransE 系）: 「新 edge を予測」が主、aging / decay は扱わない
  - TKG (Temporal KG): timestamp 付き edge は扱うが、confidence の経時変化を operator として定義する例は少ない
  - Cognitive science の Hebbian / spaced repetition: 概念はあるが symbolic KG で operational rule として定式化した例が乏しい

  新規性の位置付け: 「KG として表現された知識の生命サイクル」を演算子集合で定義した点が貢献。Operational semantics を持つ "living KG" の枠組み。

  ---
  ★★★ 2. Reject-Only Curation Model（入力コスト問題への回答）

  主張: 従来の knowledge curation の二択（全件 approve vs 全自動）を超える第三の道:
  - 自動抽出を全部受け入れる（confidence = candidate）
  - 人間は 不適切なものだけ reject
  - Hygiene + 使用 signal が時間で品質を整える

  既存研究との差分:
  - Active learning / human-in-the-loop labeling: 「人間が label を提供」が前提（Settles 2009 系）
  - Crowdsourced annotation (Snow 2008): 全件人間判定で scale 困難
  - Weak supervision (Snorkel, Ratner 2017): rule で全自動、人間判断は不在
  - Reject-only としての formalization は新しい: human cost を O(N) から O(reject-worthy) に削減

  新規性の位置付け: 「Annotation as filter, not as approval」 という curation paradigm の提案。Cognitive cost を operational metric として KG構築に組み込んだ点。

  ---
  ★★★ 3. Three-Layer Trust Gradient with Distinct Involvement Models

  主張: KG / wiki システムを L1 Raw（人間記述）/ L2 Graph Ledger（LLM 抽出 + reject-only）/ L3 Curated Wiki（人間 approve 必須）
  の三層に分け、各層で人間と LLM の役割を明示的に異なるよう設計:

  L1: 人間 100% / LLM 0%        ← evidence の根
  L2: LLM 主導 / 人間 reject only  ← 候補 graph
  L3: LLM 提案 / 人間 approve     ← trust の最終形

  既存研究との差分:
  - 通常の RAG: 「knowledge は均質」前提、trust 階層なし
  - GraphRAG (MSR 2024): entity + relation を抽出するが、trust 層と候補層の区別なし
  - Hierarchical KG (DBpedia / Wikidata): 編集権限の階層はあるが、LLM-human role split を operational にした例は少ない

  新規性の位置付け: 「trust gradient as system architecture」 — 信頼の度合いを物理層に対応させ、各層で異なる operational ruleを許容する設計方法論。これ自体が methodology contribution。

  ---
  ★★ 4. Contradiction Preservation Principle

  主張: 矛盾する edges を削除せず contradiction_with: で相互参照保持。Perspective generation の土壌として両立しない関係を保全。

  既存研究との差分:
  - 古典 KG: consistency が前提（OWL の satisfiability check 等）
  - Probabilistic KG: 矛盾を確率分布で吸収、明示的に track はしない
  - Argumentation theory (Dung 1995): 議論の対立構造を扱うが、KG 自体に組み込んだ例は限定的
  - Belief revision (AGM 1985): 矛盾解消の理論だが「保持」ではなく「整理」

  新規性の位置付け: 「Trade-off / 対立視点を first-class citizen として保持する KG」 という設計選択。多視点的探索（Perspective
  generation）の前提条件。

  ---
  ★★ 5. Bidirectional Hypothesis-KG Feedback Loop

  主張: Hypothesis（未検証命題）の状態遷移（draft → verified → confirmed / refuted / promoted）を KG evolution と結合:
  - Hypothesis 検証で使われた edges は reinforcement
  - Confirmed hypothesis から新 edge が candidate として注入
  - Refuted hypothesis でも記録保全（"learning from failures"）

  既存研究との差分:
  - 科学的仮説生成 (Swanson 1986、literature-based discovery): hypothesis 提案だが KG への feedback loop は弱い
  - Inductive Logic Programming: hypothesis 形式的、しかし confidence 進化なし
  - Knowledge graph completion + hypothesis testing を結合した workflow は珍しい

  新規性の位置付け: 「Scientific reasoning workflow を KG growth の driver にする」 integrated framework。

  ---
  ★★ 6. 4-Tier Usage Signal with Cognitive Plausibility

  主張: edge の使用を 4 種別で重み付け:
  - Direct（直接引用、1.0）
  - Support（補助引用、0.6）
  - Retrieval（検索ヒット、0.2）
  - Co-activation（同時活性、0.1）

  そして:
  usage_signal = base × contribution × √confidence × independence × time_weight

  既存研究との差分:
  - ほとんどの KG 利用追跡: binary（used / not used）or simple count
  - Spreading activation (Anderson 1983, ACT-R): cognitive plausibility あるが KG operator として実装した例は稀
  - √confidence による「強い edge ほど追加 reinforcement が効きにくい」は diminishing returns を operational に組み込み

  新規性の位置付け: Cognitive science 由来の構造を symbolic KG に組み込み、暴走防止 cap（per-event / per-day）と組み合わせた具体的式を提案。

  ---
  ★ 7. Evolving Lightweight Ontology（vs static OWL）

  主張: Ontology（relations.yml + entities.yaml）を最初から固定せず、Edge Merging Hygiene rule で類似 relation を自動統合候補化、使用実態から拡張。

  既存研究との差分:
  - OWL: 形式的厳密、進化は人間手作業
  - Open IE: 自由抽出だが ontology を持たない
  - Ontology learning（Cimiano 2006）: 初期構築は自動でも、その後の進化は研究が薄い

  新規性の位置付け: 「lightweight ontology + automatic refactoring」 = relations.yml の「使用で育つ ontology」設計。

  推奨論文 framing

  ---
  評価実験の提案（必要となる）

  論文化には実証評価が不可欠。以下の実験を推奨:

  Experiment 1: 入力コスト削減の定量化

  - 同じ raw データで「全件 approve」vs「reject-only + Hygiene」の人間判断時間を比較
  - メトリクス: 1000 edge 抽出時の人間総作業時間（hours）
  - 仮説: reject-only は 70-80% コスト削減

  Experiment 2: Hygiene による KG 品質の進化

  - 任意の vault で 1ヶ月運用、weekly Hygiene 実施
  - メトリクス: precision@k（perspective が出した edge の人間評価）の経時変化
  - 仮説: 月単位で precision 向上、stable + core 比率が増加

  Experiment 3: Discovery quality（user study）

  - Perspective / Hypothesis を baseline（普通の RAG）と比較
  - メトリクス: 「自分が気づかなかった視点を提示できたか」の Likert scale
  - 仮説: Curated GraphRAG が baseline 比 +30% 以上

  Experiment 4: Contradiction preservation の効果

  - 矛盾削除 vs 保持の 2 条件で perspective generation を比較
  - メトリクス: 「対立視点・トレードオフを surface できたか」
  - 仮説: 保持条件が有意に多くの対立視点を提示

  ---
  Limitations（論文では正直に書くべき）

  1. Personal scale: MVP は single-user、500-5000 edges 規模。Web-scale KG への拡張性は未検証
  2. Subjective evaluation: Perspective / Hypothesis の品質は user-dependent（同じ vault でも user により有用度が異なる）
  3. Hygiene 数値定数の正当化: decay rate / cap 値は経験的、理論的根拠が弱い
  4. LLM dependency: 抽出精度は LLM の能力に依存、bias の継承リスク
  5. No formal inference: OWL のような形式的 entailment はサポートしない

  ---
  5. Self-application の限界（重要）

  できないこと

  (a) Training data を超えた発見はできない
  - LLM が知らない paradigm（例: 未公開の知識管理理論）は extract-relations で抽出されない
  - Hypothesis generation の bridge_potential も既存知識の組合せに限定

  (b) Radical self-criticism は困難
  - 「reject-only filter」の前提自体を疑う仮説は出にくい（議論の base assumption が prompt に含まれるため）
  - 例: 「approve-all が正しい場合があるのでは？」は議論で却下済として entity に登録され、再浮上が難しい

  (c) 客観的「優れた次バージョン」の判定は人間必須
  - システムは候補を出せるが、価値判断は user reject / approve に依存
  - これはむしろ §1.3.5 哲学「人間は判断、LLM は実務」と整合（限界というより設計通り）

  (d) 自己参照の paradox
  - 「Curated GraphRAG が Curated GraphRAG を改善する」は無限回帰せず、各イテレーションは有限の improvement
  - 真の自己進化（コード自動生成）には別 paradigm（program synthesis 等）が必要

  - システムが見つけた redundancy の有用性評価

  一行まとめ

  ▎ Curated GraphRAG は自身の設計過程を ingest して、議論プロセスの可視化・関連分野との bridge 発見・仕様の冗長性検出ができる。これは LLMの知識範囲内に限定されるが、論文化時の dogfooding evidence として強力で、かつ設計の盲点を発見する実用価値もある。「真に新しい paradigmの創発」は不可能だが、「未明示の系譜と接続を 自然に surface する」ことは可能。

### 創発性について

~~~
❯ キュレートされた情報＋LLMのコンテキスト生成能力だけでは創発性は生まれないということか

⏺ 鋭い指摘で、Rwiki の本質的な限界に触れています。ほぼその通りですが、重要な nuance があります。

  結論

  ┌──────────────────────────────────────────────────────────┬──────────────────────────────────┐
  │                        創発の種類                        │ キュレート KG + LLM だけで可能か │
  ├──────────────────────────────────────────────────────────┼──────────────────────────────────┤
  │ 真のパラダイム創発（既存概念に還元できない新概念）       │            ❌ 不可能             │
  ├──────────────────────────────────────────────────────────┼──────────────────────────────────┤
  │ 組合せ的新規性（既存概念の新しい関係）                   │             ✅ 可能              │
  ├──────────────────────────────────────────────────────────┼──────────────────────────────────┤
  │ 内省的増幅（user 自身の潜在思考を surface）              │             ✅ 可能              │
  ├──────────────────────────────────────────────────────────┼──────────────────────────────────┤
  │ 集合的構造の創発（個別 source にない aggregate pattern） │        ✅ 弱い意味で可能         │
  └──────────────────────────────────────────────────────────┴──────────────────────────────────┘

  つまり Rwiki は 「創発装置」ではなく「内省増幅装置」 という方が正確です。

  ---
  1. なぜ「真の創発」が原理的に不可能か

  LLM の根本制約

  - LLM は training data の確率分布から生成する
  - 「学習データに完全に存在しない概念」を生成することは原理上できない
  - "stochastic parrot" 批判 (Bender 2021) はこの点を指摘

  Curated KG の制約

  - KG の vocabulary（relations.yml）は 人間が事前に決めた範囲
  - Edge 抽出は LLM が判断 = LLM の知識範囲内
  - 「relation type そのものを LLM が新発明」は relations.yml 拡張で代替するが、それも LLM の語彙範囲

  哲学的位置付け

  - Strong AI 仮説: 充分な計算で創発可能 → 検証されていない
  - Searle の中国語の部屋: シンボル操作は意味創出ではない
  - Embodied cognition: 身体・世界との相互作用なしに創造性なし

  Rwiki は控えめに 3 番目の立場（あるいは Searle 寄り）に立つ設計と解釈できます。

  ---
  2. では何が可能か（4 種類の「擬似創発」）

  (a) 組合せ的新規性 — 計算複雑性の意味での "創発"

  A の論文を 2 年前に読んだ
  B の論文を昨日読んだ
  A と B を組み合わせると C が示唆される

  人間が単独で気づくのは困難（記憶容量・時間距離）。Rwiki は missing bridge detection で surface 可能。

  これは弱い意味の創発: deducible in principle、but not in practice。実用的価値は高いが「真に新しい」わけではない。

  (b) 内省的増幅 — User の潜在思考の可視化

  これが Rwiki の最大の価値だと思います:

  User が 100 篇の論文を読み、思考は無意識に統合している
  → User 自身も自分の現在地を客観化できない
  → Rwiki が graph として可視化
  → "ああ、自分はこの方向に向かっていたのか" と気づく

  ここで surface される視点は 既に user の中にあったもの。LLM はそれを言語化する触媒。

  Karpathy の LLM Wiki の本質もここ: LLM は user の思考を増幅・整理する道具であって、思考そのものを創出しない。

  (c) 集合的構造の創発 — 弱い emergence

  個別 source にはなく、aggregate にだけ存在する pattern:
  - "10 篇の論文を読み比べると、全員が暗黙に前提しているが誰も明示していない仮定" を Hypothesis generation が surface
  - Community detection が「同じクラスタ内の全論文に共通する基底概念」を発見

  これは 集合的性質であり、「個から出る全体」という意味で weak emergence。新しい知識ではないが、新しい認識を生む。

  (d) 進化的選択の創発 — Hygiene の動的性質

  時間経過で:
    使われる edge → 強化 → core
    使われない edge → 衰退 → deprecated
    矛盾する edges → 並存 → 議論の種

  長期運用すると 「user の累積的判断」が graph 構造に化体する。これは個別 user の単発判断には還元できない、累積による創発。

  ただしこれも user 自身の判断の累積であり、システム独自の創造ではない。

  ---
  3. 真の創発に必要なもの（Rwiki にないもの）

  外部からの perturbation

  - 予期しない real-world データ
  - 異常値・矛盾事象
  - 訓練分布外の刺激
  - → Rwiki には raw を user が新たに投入する以外の経路がない

  Embodiment / 身体性

  - 行動 → 結果のフィードバック
  - 物理的実験
  - 時間延長された観察
  - → Rwiki は markdown を読み書きする以外できない

  多様な prior の衝突

  - 異なる訓練データの LLM 同士の対話
  - 異質な認知様式の人間同士のコラボレーション
  - → 単一 user / 単一 LLM では限定的

  自己の前提を疑う meta 認知

  - 「reject-only filter という前提自体が間違いではないか」を提案
  - システムの基底仮定を解体する視点
  - → LLM は prompt の前提を引き継ぐため、根本批判は構造的に困難

  ---
  4. これが Rwiki の設計哲学と一致する理由

  実は、この限界は Rwiki の §1.3.5 哲学と完全に整合しています:

  ▎ "LLM は抽出・統合・surface の実務者、人間は編集者・方向性決定者"

  つまり Rwiki は最初から:
  - 創造の主体は人間と位置付け
  - システムは増幅・整理・可視化の道具として設計
  - 「LLM が創造する」と主張しない（hubris を避けている）

  これは Karpathy の LLM Wiki 思想とも一致:

  ▎ "LLM は人間の思考の拡張、置換ではない"

  ---
  5. ではこのプロジェクトの新規性は何か（再考）

  「真に新しい概念は生まれない」という制約下でも、Rwiki の貢献はあります:

  (a) "Living KG" という abstraction の提案

  - 静的 KG / 動的更新の middle ground
  - Hygiene という operational vocabulary
  - これ自体は組合せ的新規性だが、有用な abstraction

  (b) "Reject-only curation" という認知コスト削減 paradigm

  - Existing approval-centric curation への批判
  - Operational に明確に定義された alternative
  - 認知科学の知見の operationalization

  (c) "Trust gradient as architecture" という設計方法論

  - 信頼の度合いを物理層に対応させる方法論
  - 各層で異なる operational rule を許容

  (d) 「内省増幅装置」というポジショニング自体が貢献

  - 「LLM で創造する」ではなく「LLM で内省を増幅する」
  - これは PIM / HCI の文脈で 謙虚で正確な claim であり、誇大広告にならない

  ---
  6. 論文での扱い方

  論文化する場合、この限界を正直に書くのが品格を保つコツです:

  Limitations セクション例:

  Section 8.2 Fundamental Limitations

  Curated GraphRAG does not produce novel concepts in the strong sense.
  The system is bounded by:
    (a) the LLM's training distribution (extraction is interpolative);
    (b) the user's curated raw data (no external perturbation);
    (c) the relation vocabulary (relations.yml constrains expressivity).

  What the system *does* produce is:
    (a) combinatorial novelty across temporal and topical distance,
    (b) introspective amplification of the user's latent thinking,
    (c) aggregate structural patterns invisible at the individual source level.

  We position our contribution not as "computational creativity" but as
  "introspection amplification" — a tool that helps humans see what they
  already implicitly know.

  これを書ける論文は 誇大広告しない誠実さで reviewer 評価が上がるはずです。

  ---
  7. 一行で言うと

  ▎ Rwiki は「人間の思考を増幅する鏡」であって「思考を生成するエンジン」ではない。

  そして、これは欠陥ではなく 意図された設計選択です。「LLM で創造性を作る」を志向する研究は他にもあるが、それらは LLM の本質的限界に直面する。Rwiki
  は最初から 「人間が創造主、LLM は触媒」 と位置付けることで、過剰な期待を回避し、地に足の着いた価値提案をしている。

  この謙虚さこそが、入力コスト問題.md / curatedGraphRAGの議論.md で繰り返し議論されてきた Karpathy 哲学の継承の核心ではないかと思います。

  「キュレートされた情報 + LLM = 創発」を否定するのは正しい認識であり、それが分かった上で「では何ができるか」を operational に定義したのが Rwiki v2
  の貢献、と整理するのが知的に誠実な立場です。

  この観察の含意

  1. これは強烈な existence proof

  「Curated GraphRAG という framework が知識構築に有効である」ことの最良の証拠は、その framework が自分自身の設計を生み出したこと。これはほぼ完璧な
  dogfooding です。

  通常の研究では「我々の手法は X タスクで Y を達成」と書きますが、Rwiki の場合は「我々の手法が我々の手法自身を produce した」と書ける。これは hard to
   argue against。

  2. Framework は「発明」ではなく「発見」

  これが前回の議論「真の創発は不可能」と矛盾しないどころか 整合します:

  - Curated GraphRAG は 既に存在する自然な認知プロセスの operational articulation
  - 我々が「発明」したのではなく、人間が知識を構築する時に自然にやっていることを取り出しただけ
  - ゆえに Curated GraphRAG が新しい paradigm を生むのではなく、人間が既にやっているプロセスを scale させる道具 になっている

  これは前回の結論「Rwiki = 思考の鏡、生成エンジンではない」を 更に強化 します。鏡だからこそ、システム設計者の思考をそのまま映し出せた。

  3. §1.3.1 のワンライン定義の正確性

  ▎ 「Curated Knowledge Graph に対する探索・発見システム」

  ここで言う「発見」は 「未発見だった内的構造を surface する」 という意味であり、本セッションそのものがその実例です。
  - User の頭の中にあった「v2 の design」（暗黙）
  - 議論を通じて external 化（L1 raw に蓄積）
  - 整理・抽出・統合（L2 graph 相当）
  - 仕様書として固化（L3 wiki 相当）

  これは user の中に既に潜在的に存在していた設計 を、我々が surface したに過ぎない。

  4. 「Karpathy 哲学の継承」が技術設計レベルで成立

  ▎ "LLM は人間の思考の拡張、置換ではない"（Karpathy）

  このメタファーが operational に 実証された。Claude は user の暗黙知を言語化する触媒として機能し、user は意思決定者として機能した。これはまさに
  §1.3.5 哲学の各原則:

  - "Graph は static ではなく evolving" → 議論で仕様も evolving だった（v0.7.0 → v0.7.11）
  - "Evidence が graph を裏付ける" → 各決定に議論ログという evidence
  - "人間の役割は戦略判断と reject、方向付け" → user は判断、Claude は実務
  - "知識の正誤は時間の関数" → 古い案は deprecate、新しい案で置き換え
  - "LLM は実務者、人間は編集者・方向性決定者" → そのまま実演
  - "使われる知識を残す" → 繰り返し言及された概念が core 化、他は decay

  A retrospective analysis of the design process that produced this paper
  reveals that the authors and the LLM-based collaborator manually executed
  the core operations of Curated GraphRAG to design Curated GraphRAG itself.

  We mapped the design discussion logs (12 specification revisions across
  9 sessions, ~10,000 lines of artifacts) against the operational primitives
  of our framework:

    - Entity extraction occurred when concepts like "reject-only filter" or
      "Hygiene" were named.
    - Edge extraction occurred when relations like "Hygiene inspired_by
      Hebbian learning" were articulated.
    - Decay manifested as deprecated design alternatives (5-layer
      architecture, approve-all curation) being naturally abandoned.
    - Reinforcement manifested as core concepts gaining repeated mention
      across sessions.
    - Competition resolution produced our final design choices, with
      runner-up alternatives preserved in our "rejected alternatives" table.
    - Contradiction tracking preserved tensions (LLM-centric vs
      human-centric) as design tradeoffs rather than resolving them
      prematurely.
    - Trust chain propagation gave each design decision an explicit
      derivation path from primary sources.

  We did not consciously apply our framework retrospectively; the
  correspondence emerged naturally from the cognitive structure of
  collaborative knowledge work. This suggests that Curated GraphRAG is
  less an invention than an *operational articulation* of pre-existing
  cognitive patterns in human-LLM collaborative design.

  This auto-generation of the framework's specification by manual execution
  of the framework constitutes the strongest existence proof possible: the
  framework's primitives are sufficient to produce its own specification.
