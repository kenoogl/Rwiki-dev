

# Layer1からLayer3へとキュレーションする場合、その情報がどうやってキュレートされたか、そのプロセスを可視化することに価値はあるか

~~~


⏺ 非常に重要な問いです。結論から言うと 大きな価値があり、しかも実は本セッションがその実例です。整理します。

  結論

  ┌──────────────────────────────┬──────┐
  │             観点             │ 価値 │
  ├──────────────────────────────┼──────┤
  │ 個人運用での内省支援         │ ★★★  │
  ├──────────────────────────────┼──────┤
  │ 研究・論文の reproducibility │ ★★★  │
  ├──────────────────────────────┼──────┤
  │ Self-application の素材      │ ★★★  │
  ├──────────────────────────────┼──────┤
  │ チーム協働での合意形成       │  ★★  │
  ├──────────────────────────────┼──────┤
  │ 将来の AI audit trail        │  ★★  │
  └──────────────────────────────┴──────┘

  ただし 「全プロセスを愚直に記録」では cost 爆発するため、selective な curation provenance として設計すべきです。

  ---
  1. 現状: 既に部分的にある

  Rwiki v2 が既に持っているプロセス記録:

  ┌──────────────────────────────────────────┬───────────────────────────────────────────────┬───────────────────────────────────────────────────┐
  │                   媒体                   │                記録される内容                 │                       不足                        │
  ├──────────────────────────────────────────┼───────────────────────────────────────────────┼───────────────────────────────────────────────────┤
  │ edge_events.jsonl                        │ edge の状態遷移 (created → promoted →         │ why の reasoning は最小限                         │
  │                                          │ reinforced)                                   │                                                   │
  ├──────────────────────────────────────────┼───────────────────────────────────────────────┼───────────────────────────────────────────────────┤
  │ rejected_edges.jsonl の                  │ reject の理由                                 │ reject のみ、approve / reinforcement の why       │
  │ reject_reason_text                       │                                               │ はない                                            │
  ├──────────────────────────────────────────┼───────────────────────────────────────────────┼───────────────────────────────────────────────────┤
  │ frontmatter update_history:              │ L3 page の意味的変更 summary                  │ deliberation の transcript はない                 │
  ├──────────────────────────────────────────┼───────────────────────────────────────────────┼───────────────────────────────────────────────────┤
  │ raw/llm_logs/chat-sessions/              │ rw chat 対話ログ                              │ 自動保存されるが、意思決定との紐付けが弱い        │
  ├──────────────────────────────────────────┼───────────────────────────────────────────────┼───────────────────────────────────────────────────┤
  │ evidence.jsonl                           │ edge の根拠 (raw 引用)                        │ source の WHAT は記録、curator の WHY はない      │
  └──────────────────────────────────────────┴───────────────────────────────────────────────┴───────────────────────────────────────────────────┘

  つまり**「結果」と「source」は記録されているが、「決定プロセス」は薄い**。

  ---
  2. 不足している 3 種の curation provenance

  (a) Extraction reasoning trace

  LLM が "Brunton ─authored→ SINDy" を抽出した時、
  なぜこの relation_type を選んだか、なぜ confidence をその値にしたかの
  LLM 自身の説明

  現状は edge の attributes（confidence、extraction_mode）に圧縮されているが、 「なぜ paraphrase ではなく explicit なのか」の判断 trace がない。

  (b) Approve / merge / split deliberation

  review/synthesis_candidates/sindy-summary.md を approve する際、
  user が:
  - どの edits を加えたか
  - 何を残し、何を削ったか
  - 競合案がある場合なぜこちらを選んだか

  現状は最終 wiki page と git diff だけで、deliberation の文脈は失われる。

  (c) Hypothesis verify の deliberation

  hypothesis "manifold-sindy" が confirmed と判定されたとき:
  - どの evidence を見て supporting と判断したか
  - なぜ partial ではなく confirmed だったか
  - 反論候補は本当に refuting でなく partial と判断した理由

  verification_attempts には結果だけが記録、判定の deliberationがない。

  ---
  3. 提案: Curation Provenance Layer

  案 A: Lightweight extension（推奨）

  既存構造を最小限に拡張:

  .rwiki/graph/
  ├── edges.jsonl                    (既存)
  ├── edge_events.jsonl              (既存)
  ├── evidence.jsonl                 (既存)
  ├── rejected_edges.jsonl           (既存)
  └── decision_log.jsonl             ★ 新設

  decision_log.jsonl（append-only、curation 決定の summary）:
  {"ts": "2026-04-25T10:30:00Z", "decision_id": "d_001",
   "type": "edge_extraction",
   "subject": "e_042 (sindy → koopman)",
   "reasoning": "明示的に 'Koopman は SINDy の一般化' と記述があり、explicit",
   "alternatives_considered": ["paraphrase", "inferred"],
   "outcome": "explicit, confidence 0.78",
   "context_ref": "raw/llm_logs/chat-sessions/chat-20260425-103000.md#L42-67"}

  特徴:
  - 全 decision を記録するのではなく、confidence が境界値付近、人間が介入した、矛盾があった等の 重要決定のみ 自動記録
  - context_ref で chat-sessions / review file への link を保持（重複しない）
  - Append-only で git diff 親和

  案 B: Review layer 拡張

  review/
  ├── synthesis_candidates/    (既存)
  ├── ...
  └── decisions/               ★ 新設
      ├── 2026-04-25-edge-promotion-e042.md
      └── 2026-04-25-hypothesis-confirm-h018.md

  主要決定を専用 review file として残す。Markdown で記述、frontmatter に decision_id / type / outcome / refs。

  案 C: L3 frontmatter への curation_trace: 追加

  # wiki/methods/sindy.md
  update_history:
    - date: 2026-04-24
      type: extension
      summary: "SINDy-MPC 章を追加"
      curation_trace: review/decisions/2026-04-24-add-mpc-section.md

  既存 update_history: に link を生やす形（軽量、既存構造への追加のみ）。

  推奨組合せ

  A（軽量 ledger） + C（L3 link） の組合せがベスト:
  - 重要決定のみ jsonl に記録（noise 抑制）
  - L3 から該当決定へ逆参照（trust chain と integrate）
  - 案 B（review/decisions/）は decisions が複雑な時のみ optional 生成

  ---
  4. Value 階層

  (a) 個人運用での内省支援 ★★★

  6 ヶ月後の自分への手紙:
  - 「なぜこの synthesis を作ったか」を思い出せる
  - 過去の判断 pattern を analyze して認知 bias を発見
  - spaced repetition の effect: decision 想起が知識を強化

  (b) 研究・論文の reproducibility ★★★

  Self-application study に直結:
  研究者A: "Curated GraphRAG で論文を書きました"
  Reviewer: "あなたの design decision の rationale を見せて"
  研究者A: "decision_log.jsonl が全て git history に残っています"

  Reproducibility は最近の academia で必須要求になりつつある。

  (c) Self-application の素材 ★★★

  本セッションが既にこれを実証している:

  本セッションで生成した artifacts:
  - 仕様書 (consolidated-spec.md, scenarios.md)        ← 結果
  - docs/curatedGraphRAGの議論.md 等 4 ファイル        ← curation process
  - docs/入力コスト問題.md (今日 762 行追加)            ← curation process（最新）
  - docs/論文化.md (今日 1136 行)                       ← curation process（最新）

  私たちは無意識に 「結果を spec drafts に、プロセスを docs/ に」 という分離を行ってきた。これは curation provenance を separate layer として保存
  するパターンの実例です。

  (d) チーム協働 ★★

  multi-user vault（Phase 2 候補）で:
  - 別の curator が「このページなぜ archive されたか」を追跡可能
  - 過去の議論を踏まえて新規判断ができる
  - Wikipedia の Talk page に相当

  (e) AI audit trail ★★

  将来の AI 規制（EU AI Act 等）への対応:
  - LLM が抽出した relations の reasoning trace
  - 自動 approve された edges の責任所在
  - 人間 curator の介入記録

  8. 推奨アクション

  MVP 範囲 (v2.0):
  - decision_log.jsonl を .rwiki/graph/ に追加（軽量実装）
  - 既存の edge_events.jsonl に reason と context_ref field を拡張
  - L3 frontmatter update_history: に curation_trace: link を optional 追加
  - Selective 記録の閾値を config に: confidence boundary、矛盾時、人間介入時、approve 時のみ

  Phase 2 (v0.8) 候補:
  - review/decisions/ layer の新設（複雑決定用）
  - Decision log への Hygiene 適用（古い decision の decay）
  - Visualization: graph UI で edge クリック → 該当決定にジャンプ
  - Decision log を Spec 6 perspective generation の追加 source として活用

  論文化候補:
  - 別論文として "Curation Provenance" を提案（論文 1 / 論文 2 とは独立）
  - Self-application study Section 7.5 で「decision log が無い限界」を limitation に正直記載
  - 論文 3 候補として roadmap に追加

  ---
  9. 哲学的観察

  ▎ Knowledge は 「what」+ 「why」 の二層

  現在の Rwiki v2 は「what」（edge / wiki content）と「source」（evidence）を保全しますが、「why」（curation reasoning）は薄い。

  docs/入力コスト問題.md の最新 762 行追加は、まさに 本セッションの「why」を保全する行為でした。これを framework に formalize すると、全 curation行為の why が遡れる system になります。

  これは Karpathy 哲学（「人間の判断を保全」）の更なる深化であり、§1.3.5 哲学 6 原則に 「7 番目の原則: なぜ そうしたかも知識の一部」を加える価値があるかもしれません。

  ---
  一行で言うと

Curation provenance は trust chain（縦軸）に対する直交次元（横軸）であり、Rwiki v2 が既に de facto で実践している pattern を formal に framework化する価値が高い。本セッションそのものが、その必要性と feasibility の両方を証明している。

  ご検討ください。MVP に組み込むか、Phase 2 / 別論文として残すかは設計判断です。

## 可視化

~~~
❯ MVP に組み込む方向で考えるが、そのデータを可視化することも同時に考える必要がある

⏺ 良い視点です。可視化を後付けにすると collected data が UI に向かないため、記録 schema と可視化を同時設計するのが正解です。設計の 4 軸と可視化 5
  tier を整理します。

  設計の 4 軸

  軸 1: 何を記録するか（Schema 設計）

  decision_log.jsonl の必須 fields:

  {
    "decision_id": "d_042",
    "ts": "2026-04-25T10:30:00Z",
    "decision_type": "edge_extraction | edge_reject | edge_promote | edge_unreject |
                      hypothesis_verify | synthesis_approve | page_deprecate |
                      tag_merge | split | hygiene_apply",
    "actor": "user | auto-hygiene | auto-extraction",
    "subject_refs": ["edges:e_042", "pages:wiki/methods/sindy.md", "hypotheses:h_018"],
    "reasoning": "明示的に 'Koopman は SINDy の一般化' と記述、explicit と判定",
    "alternatives_considered": ["paraphrase (rejected: 直接記述ゆえ不適)", "inferred"],
    "outcome": "explicit, confidence 0.78",
    "context_ref": "raw/llm_logs/chat-sessions/chat-20260425-103000.md#L42-67",
    "evidence_ids": ["ev_034"]
  }

  重要設計: subject_refs を 複数対象に対応（1 つの decision で edge と page 両方影響することがある）、context_ref で 議論本体
  へリンク（決定だけ重複しない）。

  軸 2: 何を記録しないか（Selective recording）

  Volume 爆発防止のため、閾値ベース自動記録:

  ┌───────────────────────────────────────────────────────────┬─────────────────────────────┐
  │                           状況                            │         自動記録？          │
  ├───────────────────────────────────────────────────────────┼─────────────────────────────┤
  │ confidence ≥ 0.85 の明確 explicit extraction              │ ❌ 通常 routine、記録しない │
  ├───────────────────────────────────────────────────────────┼─────────────────────────────┤
  │ confidence 0.45 ± 0.05（candidate-weak boundary 付近）    │    ✅ 境界決定として記録    │
  ├───────────────────────────────────────────────────────────┼─────────────────────────────┤
  │ Contradiction 検出時                                      │         ✅ 必須記録         │
  ├───────────────────────────────────────────────────────────┼─────────────────────────────┤
  │ 人間 reject / approve 時                                  │         ✅ 必須記録         │
  ├───────────────────────────────────────────────────────────┼─────────────────────────────┤
  │ Hypothesis verify                                         │         ✅ 必須記録         │
  ├───────────────────────────────────────────────────────────┼─────────────────────────────┤
  │ Hygiene routine decay / reinforcement                     │     ❌ event log で十分     │
  ├───────────────────────────────────────────────────────────┼─────────────────────────────┤
  │ Hygiene が status を遷移させた時（candidate → stable 等） │           ✅ 記録           │
  ├───────────────────────────────────────────────────────────┼─────────────────────────────┤
  │ Synthesis page を作成した時                               │           ✅ 記録           │
  └───────────────────────────────────────────────────────────┴─────────────────────────────┘

  これで decisions/day を 数十件以下 に抑制（fully automatic では数千〜万 events/day になりうる）。

  軸 3: どう Hygiene と統合するか

  3 つのオプション:

  ┌───────────────────────────────────────────────────────────────────┬──────────────────────────────────────┬──────────────┐
  │                                案                                 │                 内容                 │     推奨     │
  ├───────────────────────────────────────────────────────────────────┼──────────────────────────────────────┼──────────────┤
  │ A. decision_log は append-only、Hygiene 対象外                    │ 全履歴永続保全、最大 reproducibility │   MVP 推奨   │
  ├───────────────────────────────────────────────────────────────────┼──────────────────────────────────────┼──────────────┤
  │ B. decision_log にも decay 適用、古い decision は importance 低下 │ Volume 抑制、重要決定のみ surface    │ Phase 2 候補 │
  ├───────────────────────────────────────────────────────────────────┼──────────────────────────────────────┼──────────────┤
  │ C. decision_log の decision を後から reject 可能（誤判断の訂正）  │ self-correction、meta-learning       │ Phase 2 候補 │
  └───────────────────────────────────────────────────────────────────┴──────────────────────────────────────┴──────────────┘

  MVP は A: シンプル、reproducibility 優先。Phase 2 で B, C を検討。

  軸 4: Privacy / git 管理

  ┌───────────────────────────────────┬──────────────────────────────────────────────────────────────────────────┐
  │               設定                │                                   内容                                   │
  ├───────────────────────────────────┼──────────────────────────────────────────────────────────────────────────┤
  │ Default                           │ decision_log.jsonl は git 管理（reproducibility、self-application 素材） │
  ├───────────────────────────────────┼──────────────────────────────────────────────────────────────────────────┤
  │ .rwiki/config.yml で privacy mode │ decisions: { gitignore: true } で local-only                             │
  ├───────────────────────────────────┼──────────────────────────────────────────────────────────────────────────┤
  │ Per-decision marking              │ decision に private: true flag、export / push 時に sanitize              │
  └───────────────────────────────────┴──────────────────────────────────────────────────────────────────────────┘

  ---
  可視化 5 Tier（記録と同時設計）

  Tier 1: CLI text views（P0 必須、MVP コア）

  rw decision history <id>            # 特定 decision の詳細
  rw decision history --edge <e_id>   # 特定 edge を生んだ decisions の timeline
  rw decision history --page <path>   # 特定 page の curation timeline
  rw decision recent --since "7 days" # 直近の decisions
  rw decision stats                   # type 別集計（extraction / reject / approve...）
  rw decision contradictions          # 過去 decisions 間の矛盾
  rw decision search "<keyword>"      # reasoning 内のキーワード検索

  出力 format:
  - 短形式: 1 行 1 decision（id / ts / type / outcome）
  - 詳細: markdown table または structured text
  - JSON: --json flag で machine-readable

  Tier 2: Auto-generated Markdown timeline（P1）

  rw decision render --edge e_042 > review/decision-views/e_042-timeline.md

  生成される markdown（Obsidian で読める）:

  ---
  type: decision_view
  subject: e_042 (sindy → koopman)
  generated_at: 2026-04-25
  ---

  # Curation Timeline: e_042

  ## 2026-02-10: Created (d_001, auto-extraction)
  - **Type**: edge_extraction
  - **Reasoning**: raw/papers/x.md で "similar mathematical foundation" 記述
  - **Alternatives**: paraphrase (rejected) / inferred (rejected)
  - **Outcome**: explicit、confidence 0.58
  - **Context**: [[raw/papers/local/x.md#L42]]

  ## 2026-03-15: Reinforced (d_023, user)
  - **Type**: hypothesis_verify
  - **Reasoning**: hypothesis "manifold-sindy" 検証で supporting として機能
  - **Outcome**: confidence 0.58 → 0.71
  - **Context**: [[review/hypothesis_candidates/manifold-sindy.md]]

  ## 2026-04-18: Promoted (d_045, auto-hygiene)
  - **Type**: edge_promote
  - **Reasoning**: confidence ≥ 0.75 達成
  - **Outcome**: candidate → stable

  これは 既存の Obsidian で navigate 可能で、追加 UI 開発不要。

  Tier 3: Mermaid diagrams（P2、Tier 2 への埋め込み）

  Tier 2 の生成 markdown に mermaid を自動埋め込み:

  gantt
      title Curation Timeline: e_042
      dateFormat YYYY-MM-DD
      section Status
      candidate (created) :2026-02-10, 36d
      candidate (reinforced) :2026-03-15, 34d
      stable :2026-04-18, 7d
      section Decisions
      d_001 extraction :milestone, 2026-02-10, 0d

  議論したい設計判断

  以下、user の判断を確認したい点:

  1. Recording trigger の細かい閾値

  # .rwiki/config.yml で tunable にする提案
  decision_log:
    auto_record_triggers:
      confidence_boundary_window: 0.05  # candidate-weak 境界 ±0.05 で記録
      require_reasoning_when:
        - human_approve
        - human_reject
        - hypothesis_verify
        - synthesis_promote
      record_silent:
        - hygiene_routine_decay
        - hygiene_routine_reinforce
        - extraction_with_confidence_above: 0.85

  これで 「いつ reasoning text を user に求めるか」 が決まる。多すぎると入力 cost、少なすぎると記録不足。

  推奨 default: human-action 系は全て require、auto-action 系は境界 / 矛盾のみ。

  2. Reasoning text の入力方法

  3 オプション:

  ┌────────────────────────────────────────┬────────────────────────────────────────────────────────────┐
  │                   案                   │                             UX                             │
  ├────────────────────────────────────────┼────────────────────────────────────────────────────────────┤
  │ A. CLI で毎回 prompt                   │ rw approve 時に "理由を 1 行で入力してください: "          │
  ├────────────────────────────────────────┼────────────────────────────────────────────────────────────┤
  │ B. LLM auto-generate from chat session │ rw chat 中の対話から LLM が要約                            │
  ├────────────────────────────────────────┼────────────────────────────────────────────────────────────┤
  │ C. Optional skip with default reason   │ 入力面倒なら "approved without specific reason" を default │
  └────────────────────────────────────────┴────────────────────────────────────────────────────────────┘

  推奨: B + C の hybrid。chat session が context にある時は auto-generate、ない時は default、user が明示記述したい時は --reason "..." flag。
