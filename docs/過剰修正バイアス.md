## 問題の構造（今の方式の限界）

いまのレビューはこうなっている：

- 「問題を見つける能力」→かなり高い（subagentで補強）
- でも
- 「それ本当に直すべきか？」→判断が曖昧

実際、ログでもこういう挙動が見えてます：

- subagent追加検出 **23件**（LLM主体と同規模） 
- しかし
- **致命級は1件のみ**（価値の大半はここ）

👉 つまり：

> **検出量 ≠ 有用性**

------

# 結論（重要ポイント）

レビューシステムの質は：

> **「何件見つけたか」ではなく
>  「無駄な修正をどれだけ減らしたか」**

で決まる

------

# 必須機能（ここを最優先で入れるべき）

## 1. 「修正必要性判定レイヤ」

各指摘に対して必ずこれを付ける：

```
necessity_assessment:
  must_fix: true | false
  reason:
    - breaks_requirement
    - causes_bug
    - security_risk
    - consistency_only
    - readability_only
    - speculative
```

------

## 2. 「過剰修正（over-engineering）検出」

これがないと暴走します。

```
overengineering_risk:
  level: high | medium | low
  signal:
    - adds_new_concept
    - expands_scope
    - requires_requirement_change
    - unclear_user_value
```

特に重要：

> **「requirementsを変更しないと成立しない修正」は危険**

------

## 3. 「放置コスト vs 修正コスト」

ここが意思決定の核

```
cost_balance:
  if_ignored:
    impact: high | medium | low
  if_fixed:
    cost: high | medium | low
```

------

# シンプルな判定ルール（これだけでかなり良くなる）

## ルール1

```
impact < fix_cost → 修正不要寄り
impact > fix_cost → 修正推奨
```

------

## ルール2

```
requirementsに紐づかない → 原則修正しない
```

（例外：セキュリティ・致命バグ）

------

## ルール3

```
複数案ある → escalate
```

（今のルールと一致）

------

# 重要な観点（見落としがち）

## 「正しいが不要な修正」

これが一番多い

例：

- 明示化した方が良い → でも実害なし
- 将来の拡張性 → でも今不要
- 理想設計 → でも現仕様で十分

👉 これ全部「やらなくていい修正」

------

# dual-reviewerへの組み込み方（最小変更）

## Step 1b の後に追加

```
Step 1c: 修正必要性評価

各指摘について：

1. これを放置したら何が壊れるか？
2. 修正すると何が増えるか？
3. requirementsに紐づくか？

→ must_fix / optional / do_not_fix を判定
```

------

## adversarial subagentの役割を変更

今：

- 「間違いを探す」

これを追加：

- **「その修正、本当に必要かを否定する」**

```
For each issue, attempt to argue that it should NOT be fixed.
```

これがめちゃ効きます。

# 優先順位（あなたの質問に対する答え）

あなたの観点を踏まえると、優先度はこう変わります：

## 最優先（P0）

1. 修正必要性判定（must_fix / optional）
2. impact vs cost
3. adversarialによる「修正否定」

------

## 次点（P1）

1. overengineering検出
2. requirements紐付けチェック

------

## 後回し（P2）

1. pattern learning
2. divergence
3. 並列最適化



詰めるなら、まず **must_fix 判定を3段階にする**のがよいです。

# 1. 判定ラベル

```yaml
fix_decision:
  label: must_fix | should_fix | do_not_fix
```

## must_fix

放置すると壊れる・危険・矛盾する。

条件：

- requirement / AC を満たせない
- security / data loss / sandbox escape
- API / schema / implementation が矛盾
- failure mode が未定義で実装不能
- downstream task が作れない

## should_fix

直した方がよいが、止めるほどではない。

条件：

- 明示性不足
- test 方針が弱い
- 将来の保守性に効く
- 読み手の誤解リスクがある

## do_not_fix

正しい指摘でも、今は直さない。

条件：

- requirements にない
- scope を広げる
- 実害が曖昧
- 修正コストが高い
- ideal design に寄りすぎ

------

# 2. 判定式

各指摘に必ずこれを付ける。

```yaml
necessity:
  requirement_link: yes | no | indirect
  ignored_impact: critical | high | medium | low
  fix_cost: high | medium | low
  scope_expansion: yes | no
  uncertainty: high | medium | low
```

判定ルールはこれで十分です。

```text
critical impact → must_fix
requirement_link=yes かつ ignored_impact>=high → must_fix
scope_expansion=yes かつ criticalでない → do_not_fix or escalate
fix_cost > ignored_impact → do_not_fix寄り
uncertainty=high → escalate
```

------

# 3. subagent の役割追加

今の adversarial reviewer は「見落とし検出」が強い。
ここに **修正否定役** を足す。

```text
For each proposed fix, argue why it may not be necessary.
Classify it as must_fix, should_fix, or do_not_fix.
Prefer do_not_fix when the issue is speculative, expands scope, or lacks requirement linkage.
```

これで「指摘を増やす方向」から「不要修正を削る方向」に制御できます。

------

# 4. 出力形式

各指摘をこうする。

```yaml

severity: important

  label: must_fix
  reason:
    - security_risk

  ignored_impact: high
  fix_cost: low
  scope_expansion: no
  uncertainty: low

```

一方で、こういうものは落とす。

```yaml
issue_id: 軽-9-3
finding: test file名をより詳しく書ける
severity: minor

  label: do_not_fix
  reason:
    - readability_only

  requirement_link: no
  ignored_impact: low
  fix_cost: medium
  scope_expansion: no
  uncertainty: low
recommended_action: leave_as_is
```

特に重要なのは、**Step 1c は Step 1a/1b の後に必ず通す**ことです。
