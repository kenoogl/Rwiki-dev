# 🎯 ① miss_type（見落としの分類）

👉 **「LLMが何を見落とすか」**

## 最終提案（6分類）

```
miss_type:
  - implicit_assumption
  - boundary_leakage
  - spec_implementation_gap
  - failure_mode_missing
  - security_oversight
  - consistency_overconfidence
```

------

## それぞれの定義（ここ重要）

### 1. implicit_assumption（暗黙前提）

**定義**：
 明示されていない前提に依存しているが、それが検証されていない

例：

- 「この入力は常に正しい」
- 「この関数は成功する前提」

👉 LLMが最も苦手な領域

------

### 2. boundary_leakage（責務境界漏れ）

**定義**：
 spec / component / phase の責務が曖昧 or 越境している

例：

- Spec 3がSpec 4の責務を決めている
- Phase 5bの内容が設計に混入

👉 Rwikiで頻出

------

### 3. spec_implementation_gap（仕様と実装の乖離）

**定義**：
 仕様と実装の対応が曖昧・矛盾

例：

- アルゴリズム名と実装が一致しない
- API signatureが曖昧

👉 Step1bでよく拾うやつ

------

### 4. failure_mode_missing（失敗モード欠落）

**定義**：
 失敗時の挙動が未定義 or 不完全

例：

- retry戦略なし
- rollback不明
- timeout未定義

👉 production事故に直結

------

### 5. security_oversight（セキュリティ見落とし）

**定義**：
 攻撃経路・悪用可能性の見落とし

例：

- symlink
- prompt injection
- PATH hijack

👉 今回の致命例がここ

------

### 6. consistency_overconfidence（一貫性過信）

**定義**：
 局所的に整合しているため、問題なしと誤判断

例：

- 「一見整合しているからOK」
- グローバル矛盾を見逃す

👉 **これが今回の論文の核心**

------

# 🎯 ② difference_type（差が出た理由）

👉 **「なぜadversarialが拾えたか」**

## 最終提案（6分類）

```
difference_type:
  - assumption_shift
  - perspective_divergence
  - constraint_activation
  - scope_expansion
  - adversarial_trigger
  - reasoning_depth
```

------

## 定義

### 1. assumption_shift（前提変更）

**定義**：
 前提を変えたことで問題が顕在化

例：

- 「失敗する場合を仮定」
- 「入力が不正と仮定」

👉 adversarialの本質

------

### 2. perspective_divergence（視点の違い）

**定義**：
 レビュー観点が異なる

例：

- primary：整合性
- adversarial：攻撃者視点

------

### 3. constraint_activation（制約の顕在化）

**定義**：
 暗黙の制約を明示したことで問題発見

例：

- メモリ制約
- 実行環境制約

------

### 4. scope_expansion（スコープ拡張）

**定義**：
 レビュー範囲を広げたことで発見

例：

- cross-spec
- 他phaseとの整合

------

### 5. adversarial_trigger（攻撃的発想）

**定義**：
 「壊す視点」を導入したことで検出

例：

- injection
- misuse

------

### 6. reasoning_depth（推論深度）

**定義**：
 単純検査ではなく多段推論で発見

例：

- 逆算検査
- failureシナリオ展開

------
