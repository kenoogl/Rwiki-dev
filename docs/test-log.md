⏺ 良い問いです。現行設計を批判的に検証します。

  ソース分類の目的を整理
  現在 Rwiki でソース分類が果たしているのは 3 つの役割：
┌──────────────────┬────────────────────────────────────────────────────────────┬──────────────────────────┐
│       役割        │                                 具体的機能                   │           必須性          │

│ ① Provenance 追跡│ frontmatter source: で「どこから来たか」を記録、                │ 必須 — Rwiki の信頼性の根幹 │

├──────────────────┼─────────────────────────────────────────────────────────────┼─────────────────────────┤
│ ② 自動補完        │ infer_source_from_path でサブディレクトリから source: を自動記入 │ あれば便利、なくても可      │
├──────────────────┼────────────────────────────────────────────────────────────┼──────────────────────────┤
│ ③ 視覚的整理      │ raw/articles/ / raw/papers/ 等でディレクトリ単位で一覧可能       │ ユーザー好みの問題          │
└──────────────────┴────────────────────────────────────────────────────────────┴──────────────────────────┘

  現行設計の問題

  5 カテゴリ（articles / papers/{zotero,local} / meeting-notes / code-snippets）が 学術研究用途に偏っており、汎用的な知識管理には狭い：

  - 分類困難なコンテンツ多数:
  童話、小説、エッセイ、歴史文書、手紙、ツイート、ポッドキャスト文字起こし、講義ノート、教科書、ブログ、ニュースレター、インタビュー etc.
  - ディレクトリ名とセマンティクスの二重結合: 「どこに保存するか」と「どんな種類か」が 1 つの軸にまとめられている（童話を articles/ に入れるとsource: "web" と誤った推論がされる）
  - 硬直性: カテゴリ追加に rw_config.VAULT_DIRS と rw_utils.infer_source_from_path の両方を変更する必要

  選択肢

  案 A：分類を廃止し、source: を frontmatter 必須フィールド化

  - raw/incoming/ をフラット化、サブディレクトリ任意
  - frontmatter で source: をユーザーが明示記入（自動推論なし）
  - lint は source: が空でないことを検証
  - 利点: 柔軟、表現力が上がる、童話でも何でも扱える
  - 欠点: ユーザーの手動入力負担が増える（ただし既に title 等は必須）

  案 B：分類を「ヒント」に格下げ（現状の意味変更）

  - 既存 5 カテゴリは残すが "optional convention" と明記
  - infer_source_from_path が未知パスに対し source: "custom" または subdir 名を返すように変更
  - ドキュメントで「強制ではない」と明示
  - 利点: 破壊的変更なし、既存 Vault 互換
  - 欠点: 設計思想の曖昧化（「推奨だが強制ではない」は UX を悪化させがち）

  案 C：分類を tag で置き換え

  - raw/ はフラット
  - 分類は frontmatter tags: で多次元表現（例: tags: [paper, quantum-computing, preprint, open-access]）
  - source: は別途「情報源の URL or ファイル origin」として記録
  - 利点: 多次元分類が可能、拡張容易
  - 欠点: ディレクトリ単位の俯瞰が失われる、既存 Vault のマイグレーションが必要

  案 D：現状維持（学術用途に特化する方針を明示）

  - 「Rwiki は学術・研究知識管理ツール」と位置づけ
  - 童話等は対象外、サンプルは削除 or 移動
  - 利点: スコープ明確、設計ブレなし
  - 欠点: 応用範囲が狭まる

  私の見解

  source: 自体は trust chain の根幹なので必須。しかし現行の「ディレクトリ名 ⇒ source 値」の自動マッピングは過度に学術文脈に寄りすぎています。

  一番クリーンなのは 案 A（分類廃止 + frontmatter 必須化） です：

  - Rwiki の核心は「raw→review→wiki の人間承認パイプライン」と「evidence-based knowledge」であり、ディレクトリ分類ではない
  - source: は「ウェブ URL」「Zotero ID」「論文 DOI」「会議日時」「本のタイトル」等多様な意味を持ちうる自由文字列にすべき
  - 自動推論は便利だが、意味的な誤りを生む（童話＝web のような）

  案 A は 破壊的変更なので、spec 起票が必要です（source-classification-refactor 的な）。ただし既存の 5 カテゴリは残したまま、「推奨パターンの 1つ」に格下げする形で後方互換を保てます。

  - 学術特化 → 案 D（現状維持 + 明示）

❯ そもそもソース分類は必要か

⏺ 良い問いです。現行設計を批判的に検証します。

  ソース分類の目的を整理

  現在 Rwiki でソース分類が果たしているのは 3 つの役割：

┌───────────────────┬───────────────────────────────────────────────────────────┬──────────────────────────┐
│       役割        │                        具体的機能                            │           必須性          │
├───────────────────┼───────────────────────────────────────────────────────────┼──────────────────────────┤
│ ① Provenance 追跡 │ frontmatter source: で「どこから来たか」を記録、               │ 必須 — Rwiki の信頼性の根幹 │
│                   │ evidence/trust chain の基盤                                │                          │
├───────────────────┼───────────────────────────────────────────────────────────┼──────────────────────────┤
│ ② 自動補完        │ infer_source_from_path でサブディレクトリから source: を自動記入 │ あれば便利、なくても可      │
├───────────────────┼───────────────────────────────────────────────────────────┼──────────────────────────┤
│ ③ 視覚的整理       │ raw/articles/ / raw/papers/ 等でディレクトリ単位で一覧可能      │ ユーザー好みの問題          │
└───────────────────┴───────────────────────────────────────────────────────────┴──────────────────────────┘

  現行設計の問題

  5 カテゴリ（articles / papers/{zotero,local} / meeting-notes / code-snippets）が 学術研究用途に偏っており、汎用的な知識管理には狭い：

  - 分類困難なコンテンツ多数:
  童話、小説、エッセイ、歴史文書、手紙、ツイート、ポッドキャスト文字起こし、講義ノート、教科書、ブログ、ニュースレター、インタビュー etc.
  - ディレクトリ名とセマンティクスの二重結合: 「どこに保存するか」と「どんな種類か」が 1 つの軸にまとめられている（童話を articles/ に入れると
  source: "web" と誤った推論がされる）
  - 硬直性: カテゴリ追加に rw_config.VAULT_DIRS と rw_utils.infer_source_from_path の両方を変更する必要

  選択肢

  案 A：分類を廃止し、source: を frontmatter 必須フィールド化

  - raw/incoming/ をフラット化、サブディレクトリ任意
  - frontmatter で source: をユーザーが明示記入（自動推論なし）
  - lint は source: が空でないことを検証
  - 利点: 柔軟、表現力が上がる、童話でも何でも扱える
  - 欠点: ユーザーの手動入力負担が増える（ただし既に title 等は必須）

  案 B：分類を「ヒント」に格下げ（現状の意味変更）

  - 既存 5 カテゴリは残すが "optional convention" と明記
  - infer_source_from_path が未知パスに対し source: "custom" または subdir 名を返すように変更
  - ドキュメントで「強制ではない」と明示
  - 利点: 破壊的変更なし、既存 Vault 互換
  - 欠点: 設計思想の曖昧化（「推奨だが強制ではない」は UX を悪化させがち）

  案 C：分類を tag で置き換え

  - raw/ はフラット
  - 分類は frontmatter tags: で多次元表現（例: tags: [paper, quantum-computing, preprint, open-access]）
  - source: は別途「情報源の URL or ファイル origin」として記録
  - 利点: 多次元分類が可能、拡張容易
  - 欠点: ディレクトリ単位の俯瞰が失われる、既存 Vault のマイグレーションが必要

  案 D：現状維持（学術用途に特化する方針を明示）

  - 「Rwiki は学術・研究知識管理ツール」と位置づけ
  - 童話等は対象外、サンプルは削除 or 移動
  - 利点: スコープ明確、設計ブレなし
  - 欠点: 応用範囲が狭まる

  私の見解

  source: 自体は trust chain の根幹なので必須。しかし現行の「ディレクトリ名 ⇒ source 値」の自動マッピングは過度に学術文脈に寄りすぎています。

  一番クリーンなのは 案 A（分類廃止 + frontmatter 必須化） です：

  - Rwiki の核心は「raw→review→wiki の人間承認パイプライン」と「evidence-based knowledge」であり、ディレクトリ分類ではない
  - source: は「ウェブ URL」「Zotero ID」「論文 DOI」「会議日時」「本のタイトル」等多様な意味を持ちうる自由文字列にすべき
  - 自動推論は便利だが、意味的な誤りを生む（童話＝web のような）

  案 A は 破壊的変更なので、spec 起票が必要です（source-classification-refactor 的な）。ただし既存の 5 カテゴリは残したまま、「推奨パターンの 1
  つ」に格下げする形で後方互換を保てます。

  ---
  この論点は Rwiki の方向性（学術特化 vs 汎用知識管理）に関わるので、どちらに振るかを先に決めるべきです。

  - 学術特化 → 案 D（現状維持 + 明示）
  - 汎用化 → 案 A（分類廃止）

  1. 根本前提への反論

  反論 1-1: 童話は本当に Rwiki のスコープか？

  - ユーザーの問題意識は「童話が articles/ に合わない」から始まった
  - しかし Rwiki は .kiro/steering/product.md 上**「研究知識管理システム」として設計**
  - 童話はスコープ外と明示すれば、そもそも案 C への動機が消える
  - 「サンプルとして置いただけ」であれば、本番運用では出現しない問題

  反論への反論: 学術分野でも対応しきれないケースがある（歴史文書・手紙・文学作品レビュー・分野横断の随想）。童話は症状であって病因ではない。

  反論 1-2: 「source = category」は本当に悪か？

  現在の source: "web" / "zotero" / "local" / "meeting" / "code" は粗い粒度の provenance として機能している：

  - source: "web" = 「ウェブ由来」というメタ情報
  - source: "meeting" = 「会議由来」というメタ情報

  粒度が粗いだけで、間違いではない。

  **案 C が主張する「童話を articles/ に入れると source: web が誤り」**は、厳密には source: web
  の意味を「ウェブから来た」と解釈するから誤りに見えるだけ。「web 層の知識」（手紙・散文・ウェブ記事を含む広い意味）と解釈すれば矛盾しない。

  反論 1-3: 問題は「分類」ではなく「AGENTS の出力形式固定」では？

  元々の事象：ユーザーが物語構造（初期状態/選択/制約...）を求めたが synthesize_logs.md の形式が固定だった。

  これは分類の問題ではなく、出力タスクの種類の問題。案 C（フラット raw + タグ）にしても、synthesize
  タスクの出力形式は固定のまま。物語構造抽出をやりたいなら、案 C とは独立に新 AGENT narrative_extract が必要。

  ⇒ 案 C は元の不満を解決しない可能性がある。

  2. タグシステム特有の既知問題

  タグベース分類が多くのシステムで破綻してきた理由：

  問題 2-1: タグの乱立・揺らぎ

  実プロジェクトで必ず発生する：

  tags: [ml]           ← ユーザー A
  tags: [machine-learning]   ← ユーザー B（または後日の本人）
  tags: [ML]           ← ユーザー C
  tags: [MachineLearning]    ← ユーザー D

  同じ概念に 4 種類のタグが並立。controlled vocabulary を強制しないと必ず起こる。

  問題 2-2: タグの意味ドリフト

  時間経過で同じタグが違う意味に使われ始める：

  - 2026-04: tags: [paper] = 論文
  - 2027-01: tags: [paper] = ブログ記事（「paper 形式のドキュメント」という拡大解釈）

  カテゴリベースならディレクトリが物理的に分離されていて、意味の拡大解釈が起きにくい。

  問題 2-3: タグの付け忘れ・網羅性欠如

  - 5 個のタグをつける「べき」コンテンツに 1 個だけつけるユーザーが多数
  - 検索時に「paper タグだけ検索」では preprint だけついた論文が漏れる
  - カテゴリなら「ディレクトリに存在する」が絶対基準

  問題 2-4: タグ付けの認知負荷

  - カテゴリ: 「これは論文／記事／会議メモのどれ？」の 1 回判断
  - タグ: 「content type / domain / status / source type の各軸で何をつける？」の多軸判断
  - 80% のユースケースではカテゴリの方が楽

  6. 代替案（案 C の refinement）

  代替 6-1: 段階的移行（2-phase migration）

  一気に破壊せず、段階を踏む：

  Phase 1: サブディレクトリを optional 化
  - VAULT_DIRS から raw/incoming/articles 等を外す（rw init で作らない）
  - infer_source_from_path は維持（既定サブディレクトリが「推奨パターン」として残る）
  - lint は source: が無い場合のみ path から推論、ある場合は frontmatter 優先
  - ユーザーはサブディレクトリを使うも使わないも自由

  Phase 2: path 推論を削除
  - infer_source_from_path 削除
  - source: が無ければ WARN（FAIL ではない）

  利点: 破壊的変更を避けつつ、タグベースへ移行できる

  代替 6-2: 設定ファイル化（.rwiki/categories.yml）

  # .rwiki/categories.yml
  categories:
    - name: articles
      source_value: web
    - name: papers/zotero
      source_value: zotero
    - name: fairy-tales        # ユーザー追加
      source_value: fairy-tale
    - name: essays
      source_value: essay

  infer_source_from_path がこの設定を読む。

  利点: 既存モデルを壊さず拡張性を得る、ディレクトリベースのメリット維持

  代替 6-3: 併用（カテゴリ + タグの両立）

  - カテゴリ（ディレクトリ）は残す → Obsidian フォルダツリー機能維持
  - tags: を追加の多次元分類として推奨
  - source: はディレクトリ推論、tags: は明示記入
  - 童話は raw/incoming/narratives/ のような新カテゴリを正式化

  利点: 後退なし、童話問題も解決

  8. 私の批判的再評価

  正直に言うと、案 C は魅力的だが以下の理由で慎重になるべき：

  1. 元の問題（物語構造抽出）は案 C では解決しない。別タスク narrative_extract が必要
  2. タグ規律の制度がないまま移行すると、1 年後に「タグが乱立して使い物にならない」状態になりがち
  3. Obsidian ユーザーの UX が後退する可能性（フォルダツリーナビの喪失）
  4. エッジケース（衝突・マルチソース）の対応コストが現状より大きい
  5. スコープが「研究知識管理」のままなら、5 カテゴリで十分

  代替として有力なのは 6-3（カテゴリ + タグ併用）：
  - 既存の堅牢性を維持
  - 多次元分類の柔軟性を獲得
  - 童話は新カテゴリ raw/incoming/narratives/ を追加するだけで対応
  - 破壊的変更なし

  もしくは 案 D（現状維持 + スコープ明示）：
  - Rwiki は学術・研究用途と明記
  - 童話は「動作確認サンプルのみ、本番運用対象外」と明記
  - 最小コスト
