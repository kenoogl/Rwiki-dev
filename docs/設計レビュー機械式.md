# 設計書のレビュー方式



## 設計書レビュートライアル

##### 設計書のレビュー方法を変えてみようと思う。今あるものは保存しておいて、比較の為に利用する。次の方針でプロンプトを考えて。

   

- 意図や仕様レベルならまだしも、設計、タスクになると細かすぎて判断のしようがないことが多い。判断負荷が大きくなりすぎ、逆に品質が低下する懸念もある。
- 設計・タスク層では、LLMに「良し悪し」を判定させるより、上位仕様へのトレーサビリティがあるか、責務が過剰に分割されていないか、完了条件が実行可能か、依存関係が循環していないか、テスト可能な単位になっているか、だけを見るのが現実的。
- つまり、設計・タスクレビューは、内容の妥当性レビューではなく、作業単位として健全かを見る。
- 特にタスクは、このタスクは何を変えるか、どの仕様に対応するか、完了条件は何か、検証方法は何か、失敗したら何が壊れるかの観点。
- 機械検証に落としていく。

#### 基本方針

設計レビュー
  ↓
構造化
  ↓
検査項目化
  ↓
機械チェック
  ↓
リスク判定

1. #### 設計書を機械可読にする

id: design.review_pipeline
type: design
parent_specs:

  - spec.review.risk_scoring
    components:
  - review_linter
  - semantic_reviewer
  - risk_classifier
    inputs:
  - spec_diff
    outputs:
  - review_report
    dependencies:
  - llm_api
  - vector_db
    failure_modes:
  - false_negative
  - excessive_escalation
    tests:
  - test_missing_spec_mapping
  - test_risk_level_required

ここで重要なのは、文章設計書の全文を判定しないことです。
 **設計書から検査用メタデータを抜き出す**のが第一段階です。

2. #### チェックを5種類に分ける

##### A. 構造チェック

機械で確実にできます。

- parent_specs が存在するか
- inputs / outputs が空でないか
- component が定義されているか
- dependency が未定義でないか
- tests が紐づいているか

##### B. トレーサビリティチェック

Spec → Design → Task → Test　がつながっているかを見る。

spec.review.risk_scoring
  ↓ implemented_by
design.review_pipeline
  ↓ decomposed_into
task.add_risk_classifier
  ↓ verified_by
test.risk_level_required

##### C. 型チェック

関係の型が正しいかを見る。Spec implements Taskはおかしい

正しくは：Design implements Spec
Task implements Design
Test verifies Spec / Task

##### D. リスクパターンチェック

設計に含まれる危険パターンを検出します。

- 単一コンポーネントが複数責務を持つ
- 入力はあるが出力がない
- 状態変更があるのに rollback がない
- 外部API依存があるのに failure_mode がない
- LLM判断があるのに confidence / escalation がない
- 自動承認があるのに human gate がない

##### E. 意味的ドリフトチェック

ここだけLLMを使います。

上位仕様:
危険な仕様変更を見逃さないため、risk level A/D は人間レビュー必須とする。

設計:
LLM が low risk と判断した変更は自動承認する。

これは構造上つながっていても意味がズレています。

この差分をLLMに検出させる。

3. #### 判定を点数化する

   設計レビュー結果は approve / reject にしない。

review_result:
  structural_score: 0.95
  traceability_score: 0.80
  type_score: 1.00
  risk_pattern_score: 0.60
  semantic_alignment_score: 0.70
  risk_level: B
  human_review_required: true

```

  パイプライン構造提案

  [design.md / tasks.md]
          ↓
  Phase 1: メタデータ抽出 (LLM)        → design_metadata.yaml
          ↓
  Phase 2-A: 構造チェック (script)     → findings[]
  Phase 2-B: トレーサビリティチェック (script)
  Phase 2-C: 型チェック (script)
  Phase 2-D: リスクパターンチェック (script)
          ↓
  Phase 2-E: 意味的ドリフト検出 (LLM)   → findings[]
          ↓
  Phase 3: 点数化 + risk_level 判定 (script)  → review_report.yaml

  LLM 呼び出しは 2 箇所のみ (メタデータ抽出 + 意味的ドリフト)。残りはすべて Python script (PyYAML + 単純判定) で機械検証。LLM
  判断回数を減らすことが「判断負荷低下 = 品質維持」の核。

  Phase 1: メタデータ抽出プロンプト

  これが新方式の中核 (構造化されたメタデータが取れれば残りはほぼ機械処理)。サンプルプロンプト:

  あなたは設計書の機械可読メタデータを抽出する extractor
  である。判断・評価・改善提案は一切行わない。設計書から事実のみを構造化して抽出することが任務である。

  # 入力
  - {design_path}: 設計書 markdown
  - {requirements_path}: 上位仕様書 markdown (parent_specs 解決用)

  # 出力形式 (YAML)

  ```yaml
  feature_name: <spec name>
  extraction_warnings: []   # 抽出不能だった項目のみ列挙、判断は書かない

  design_units:             # 設計書内の各「設計対象」を 1 unit として抽出
    - id: design.<snake_case_id>     # design.md 内 component / module / data model / flow から命名
      type: component | module | data_model | flow | interface
      location: <file:line>          # 抽出元の行番号
      parent_specs:                  # 設計書本文に明示記載された AC ID (例: 5.2) のみ。推測しない
        - <numeric_id>
      components: [<name>, ...]      # この unit が含む下位 component (なければ空)
      inputs: [<param>, ...]         # signature / API contract から抽出 (なければ空)
      outputs: [<return>, ...]       # 同上
      dependencies:                  # Allowed Dependencies / Outbound dependency 明示
        - kind: internal | external | spec
          target: <name or spec_id>
      failure_modes: [<name>, ...]   # Failure Conditions / Error Handling 明示分のみ
      state_change: bool             # 状態変更があるか (write 操作 / persistence)
      rollback_defined: bool         # 状態変更時の rollback / cleanup 記述があるか
      llm_judgment: bool             # LLM 呼出を含むか
      llm_confidence_or_escalation: bool   # 信頼度 / human gate / escalation 記述があるか
      auto_approval: bool            # 自動承認系の振る舞い
      human_gate: bool               # 人間判断 gate
      tests:                         # Testing Strategy / Test Plan で名指しされたテスト
        - <test_name>

  requirements_index:         # requirements.md から AC ID 一覧を抽出
    - id: <numeric_id>
      summary: <1 行>
      location: <file:line>

  cross_references:           # 設計書本文に書かれた spec → design / task → design 等の関係
    - from: <id>
      to: <id>
      relation: implements | decomposed_into | verifies | depends_on
  ```

  # ルール
  - 抽出元の「明示記述」のみを使う。設計書に書かれていないことは推測しない。
  - 値が欠けている項目は省略でなく `null` または空配列で明示。Phase 2 構造チェックが欠落を検出する。
  - judgement / 改善提案 / 良し悪し評価は出力に含めない (これは別フェーズの責務)。
  - 抽出が不能な箇所 (例: signature が text のみで構造不明) は `extraction_warnings` に「unit_id + 理由」のみ記録。

  このプロンプトは判断を絶対に含めない設計です — extractor が「これは責務多すぎ」とか書き始めると新方式の意義が崩れる。判断は Phase 2 機械 script +
  Phase 2-E LLM の役割に閉じ込める。

  Phase 2-A〜D: 機械検証 script 仕様

  Python pseudocode (実装は別タスク):

  def check_structural(metadata: dict) -> list[Finding]:
    # parent_specs / inputs / outputs / components / dependencies / tests の空欠落
    ...
  def check_traceability(metadata: dict, requirements: dict) -> list[Finding]:
    # design.parent_specs 各 ID が requirements_index に存在
    # requirements_index 各 ID が少なくとも 1 design に implemented_by
    # design unit に test 紐付けあり
    ...
  def check_type(cross_references: list) -> list[Finding]:
    # 許容 relation 集合: {Design implements Spec, Task implements Design, Test verifies Spec|Task}
    # それ以外 (例: Spec implements Task) は ERROR
    ...
  def check_risk_patterns(metadata: dict) -> list[Finding]:
    # responsibilities >= 3, inputs without outputs, state_change without rollback,
    # external_dep without failure_modes, llm_judgment without confidence/escalation,
    # auto_approval without human_gate
    ...

  各 check は [Finding(severity, location, detail)] を返す。score 算出は 通過件数 / 全件数。

  Phase 2-E: 意味的ドリフト検出プロンプト

  LLM の役割は「上位仕様の意図と、設計の振る舞いがズレているか」のみに絞る:

  あなたは上位仕様と設計の意味整合性のみを検査する。構造的妥当性 / 細かな改善提案 / コードスタイルは検査しない。

  # 入力
  - {requirements_excerpts}: requirements.md から抽出した「制約・規範・禁止事項」を表す文 (Phase 1 metadata で明示参照された AC のみ)
  - {design_excerpts}: 設計書から抽出した「振る舞い・自動判断・条件分岐」記述

  # タスク
  各 requirement_excerpt と、それを implemented_by する design_excerpt を 1 ペアずつ比較し、以下のいずれかに該当するか判定する:

  1. drift_authority: 上位が「人間判断必須」と書いているのに、設計が「LLM 自動判断で進む」になっている
  2. drift_conservatism: 上位が「危険側に倒す」と書いているのに、設計が「効率側に倒す」になっている
  3. drift_scope: 上位が規定する適用範囲と、設計の適用範囲がズレている
  4. drift_invariant: 上位の不変条件 (atomicity / consistency / monotonicity) を、設計が破る経路を含む
  5. no_drift: 構造上も意味上も整合

  # 出力形式 (YAML)
  findings:
    - pair_id: <requirement_id>::<design_unit_id>
      drift_kind: drift_authority | drift_conservatism | drift_scope | drift_invariant | no_drift
      severity: ERROR | WARN | INFO
      excerpt_requirement: <quoted text>
      excerpt_design: <quoted text>
      explanation: <1-2 文、「意図と異なる経路はこれ」を客観的に記述>

  # ルール
  - 「もっと良い設計はある」「ここを改善すべき」は書かない。意図と設計のズレのみを記述。
  - ズレが見当たらない場合は drift_kind: no_drift で記録 (省略しない)。
  - 推測で drift を捏造しない。requirement / design に明示記載がない場合は no_drift。

  LLM が「内容批評」に逸脱しないよう、禁止事項を明文化するのがポイント。

  Phase 3: 採点ロジック

  review_result:
    feature_name: <name>
    scores:
      structural_score: 通過件数 / 全件数
      traceability_score: 同上
      type_score: 同上
      risk_pattern_score: 同上
      semantic_alignment_score: (drift != no_drift) 件数を全 pair で減点
    composite_score: weighted_avg(...)   # 重み: trace 0.3 / risk 0.25 / structural 0.2 / type 0.15 / semantic 0.1 (調整可)
    risk_level: A|B|C|D
    human_review_required: bool
    findings: [...]                       # 全 phase の findings 統合

  risk_level 判定ロジック (素朴版):
  - ERROR 0 件 + 全 score >= 0.95 → A (人間レビューなしで approve 可)
  - ERROR 1+ 件 または semantic_alignment_score < 0.8 → D (人間判断必須)
  - それ以外 score 範囲で B / C

  human_review_required は risk_level B 以上で常に true、A のみ自動 approve 可能。

  確認したいこと

  1. 適用試行先: 直前にコミットした Spec 2 design でこの新方式をドッグフードするでよいか? (旧方式 10 ラウンドは Spec 5 で完了済、Spec 2
  で新旧比較できる構図)
  2. memory 整理: 新方式は feedback_design_review_mechanical.md として追加 + 旧 5 件は維持 + MEMORY.md に「設計レビュー方法論
  v2」セクション追加、で進めて OK?
  3. 詳細化スコープ: 上記 5 段階のうち、まず Phase 1 メタデータ抽出プロンプトを完成させる優先度が一番高いと思うが、Phase 2 script の Python
  実装まで本セッションでやるか? それとも仕様だけ確定して実装は次セッション以降?

  まず 1 と 2 を決めてから、3 の粒度を決めるのが効率的だと思います。
```
